# 62 - Feature (Tracks-Player): Live-Status im Hauptkanal + Schutz vor Fehlmanipulation

## 1. Why

### 1.1 Objective

Der DJ möchte in der Tracks-/Playlist-Tabelle auf einen Blick sehen, welcher Track gerade im
Hauptkanal (Dancefloor) läuft, und ihn nicht versehentlich per Klick stoppen oder durch einen
anderen Track ersetzen können — ein Fehlklick während eines laufenden Sets ist heikel.

### 1.2 Context

CLAUDE.md dokumentiert bisher bewusst das Gegenteil: "Row buttons always show '▶' — play/pause
state is only ever shown in the global bar, avoiding needing to sync state across every row." Für
den Cue-/Vorhörkanal wurde dieses Prinzip in Intent 51 (Nachtrag) schon einmal bewusst durchbrochen
— der Vorhör-Button färbt sich rot mit Pause-Symbol, während sein Track im Cue-Kanal läuft, per
`cue-player:state`-Event (`cue_player_controller.js` broadcastet, `audio_trigger_controller.js`
hört zu). Dieser Intent überträgt dasselbe Muster auf den Haupt-Play-Button (grün statt rot) und
ergänzt zusätzlich einen Schutz-Mechanismus, den der Cue-Kanal nicht braucht (Vorhören ist per
Definition unterbrechungsfrei für den Dancefloor, der Hauptkanal dagegen nicht).

### 1.3 Depends On Intents

* 40 - Feature (Tracks-Player): Persistenter Audio-Player
* 51 - Feature (Tracks-Player)/Nachtrag: Cue-/Vorhörkanal, Live-Row-State-Muster

## 2.0 What

```gherkin
Feature: Tracks-Player - Live-Status im Hauptkanal + Schutz vor Fehlmanipulation

  Als DJ
  möchte ich sehen, welcher Track gerade im Hauptkanal läuft
  und ihn nicht versehentlich stoppen oder durch einen anderen ersetzen können

  Scenario: Zeilen-Button färbt sich grün mit Pause-Symbol, während der Track spielt
    Given ich starte einen Track über seinen Zeilen-Button
    Then zeigt genau dieser Button ein Pause-Symbol mit grünem Hintergrund

  Scenario: Klick auf den aktiven Button fragt vor dem Pausieren nach
    Given ein Track spielt, sein Zeilen-Button zeigt das Pause-Symbol
    When ich erneut auf diesen Button klicke und bestätige
    Then pausiert die Wiedergabe

  Scenario: Ein abgebrochener Pausier-Versuch ändert nichts
    Given ein Track spielt
    When ich erneut auf seinen Button klicke und die Bestätigung abbreche
    Then spielt der Track unverändert weiter

  Scenario: Klick auf einen anderen Track während laufender Wiedergabe fragt nach
    Given Track A spielt
    When ich auf den Zeilen-Button von Track B klicke und bestätige
    Then wechselt die Wiedergabe zu Track B

  Scenario: Ein abgebrochener Wechsel lässt den laufenden Track unverändert
    Given Track A spielt
    When ich auf den Zeilen-Button von Track B klicke und die Bestätigung abbreche
    Then spielt weiterhin Track A

  Scenario: Play ohne laufende Wiedergabe fragt nicht nach
    Given kein Track spielt gerade im Hauptkanal
    When ich einen Zeilen-Button anklicke
    Then startet die Wiedergabe sofort ohne Bestätigungsdialog

  Scenario: Live-Status übersteht eine Seitennavigation
    Given ein Track spielt, sein Button zeigt grün
    When ich auf eine andere Seite navigiere und zurück
    Then zeigt der Button weiterhin den grünen Live-Zustand
```

## 3.0 How

### 3.1 Implementation Context

Rein clientseitig, kein Server-Code betroffen. Exaktes Vorbild: der bestehende Cue-Kanal-
Live-State-Mechanismus (Intent 51 Nachtrag).

* **`audio_player_controller.js`**: neue `broadcastState()`-Methode, aufgerufen aus den
  bestehenden `play`/`pause`-Listenern (plus `ended`) auf `audioTarget` — dispatcht
  `audio-player:state` (`{ url, playing }`) auf `document`, analog zu
  `cue_player_controller.js#broadcastState`.
* **`audio_trigger_controller.js`**: `mode: "play"` bekommt dieselbe `connect`/`disconnect`/
  `turbo:load`-Sync-Logik wie `mode: "cue"` (Feld `isActivePlayTrack`, `applyPlayState(url,
  playing)` färbt `btn-success` statt `btn-outline-secondary` und zeigt `⏸` statt `▶` — Grün statt
  Rot, sonst identisches Muster inkl. Sync auf `turbo:load` wegen der bekannten Permanent-Element-
  Timing-Falle).
* **`play()`** erweitert um die Schutzlogik (liest den Hauptkanal-Zustand direkt und live per
  `document.querySelector('[data-audio-player-target="audio"]')`, kein zusätzlicher State nötig):
  - Ist der eigene Button gerade aktiv (`isActivePlayTrack`): `confirm(...)` vor dem Pausieren:
    bei Bestätigung ein neues `audio-player:toggle`-Event dispatchen (der globale Player braucht
    dafür nur einen Listener, der seine bestehende `toggle()`-Methode aufruft — dieselbe, die der
    Player-eigene Play/Pause-Button schon nutzt).
  - Ist der Hauptkanal-`<audio>` vorhanden und `!paused`, aber ein anderer Track: `confirm(...)`
    vor dem Wechsel; bei Bestätigung wie gehabt `audio-player:play` dispatchen.
  - Sonst (nichts spielt): sofort abspielen wie bisher, kein Dialog.
* System-Specs steuern die nativen `confirm()`-Dialoge über Capybaras `accept_confirm`/
  `dismiss_confirm` (Cuprite unterstützt das JS-Dialog-API).
* CLAUDE.md-Abschnitt "Row buttons always show '▶'" wird aktualisiert — die Regel galt bis zu
  diesem Intent, jetzt gilt für den Play-Button dasselbe Live-Status-Prinzip wie für den
  Vorhör-Button.
