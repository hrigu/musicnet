# 71 - Feature (Playlists): Farbkonzept für Badges + freie Farbwahl

## 1. Why

### 1.1 Objective

Manche Playlist-Badges sind fast weiss und dadurch kaum erkennbar. Der DJ möchte ausserdem, dass
Playlists mit ähnlichem Namen ähnlich eingefärbt werden (Wiedererkennung auf einen Blick), und dass
sich die Farbe pro Playlist auf deren Detailseite frei wählen lässt (Override).

### 1.2 Context

Root Cause des Kontrast-Bugs: `PlaylistsHelper::CONTEXT` enthält `"bg-light"` als reine
Bootstrap-Hintergrundfarbe ohne das zugehörige `text-bg-light` (das automatisch für dunklen,
lesbaren Text sorgt) - kombiniert mit der weissen Standard-Textfarbe von Bootstraps `.badge`
ergibt das einen fast weissen Text auf fast weissem Grund. `genre_badge`
(`tracks_helper.rb`) macht es bereits richtig (`text-bg-light`) - `playlist_color_class` nicht.

Die Farbzuordnung basiert aktuell auf einem Checksum über `playlist_short_name` (Name nach
Whitespace-Entfernung und "fusion"/"blues"-Kürzung) - zwei ähnlich benannte, aber nicht identische
Playlists (z.B. "Fusion Montag" und "Fusion Dienstag") landen dadurch praktisch immer auf
unterschiedlichen Farben, da der volle (gekürzte) String einfliesst, nicht nur ein gemeinsamer
Namensteil.

DJ-Gig-Playlists (Name beginnt mit einer 4-stelligen Jahreszahl, `dj_playlist?`) haben bereits
einen eigenen Sonderfall mit fester Farbe (`bg-dark`) - laut DJ ist genau das gewünscht (alle
eigenen Gigs sollen dieselbe Farbe behalten) und bleibt unverändert, nur der Klassenname wechselt
mit auf `text-bg-dark` für Konsistenz.

Der Badge-Markup-Block (`<span class="badge <%= playlist_color_class(...) %>">...</span>`) ist an
fünf Stellen dupliziert (`playlists/_playlist.erb`, `tracks/_track.erb`, `tracks/show.html.erb`,
`artists/_artist.erb`, `playlist_tracks/_playlist_track.erb`) - eine zentrale Helper-Methode für
das komplette Badge-Markup vermeidet, dass die neue Override-Logik an mehreren Stellen dupliziert
werden muss.

### 1.3 Depends On Intents

* 24 - Feature (Playlists): führte `playlist_color_class`/`playlist_short_name` ein.
* 43 - Feature (Tracks-Suche): `genre_badge` als Vorbild für korrekten `text-bg-light`-Kontrast.

## 2.0 What

```gherkin
Feature: Playlist-Badges - Farbkonzept + freie Farbwahl

  Als DJ
  möchte ich Playlist-Badges immer lesbar sehen, ähnlich benannte ähnlich eingefärbt
  und die Farbe pro Playlist frei wählen können
  damit ich Playlists auf einen Blick erkenne und unterscheide.

  Scenario: Kein fast-weisses Badge mehr
    Given eine Playlist bekommt automatisch eine helle Farbe zugewiesen
    Then ist der Badge-Text trotzdem klar lesbar (ausreichender Kontrast)

  Scenario: Ähnlich benannte Playlists gleich eingefärbt
    Given zwei Playlists "Fusion Montag" und "Fusion Dienstag"
    Then haben beide dieselbe automatische Badge-Farbe

  Scenario: DJ-Gig-Playlists bleiben einheitlich
    Given mehrere Playlists, deren Name mit einer Jahreszahl beginnt
    Then haben alle dieselbe Badge-Farbe, unabhängig vom Rest des Namens

  Scenario: Freie Farbwahl auf der Detailseite
    Given ich bin auf der Bearbeiten-Seite einer Playlist
    When ich eine eigene Farbe wähle und speichere
    Then erscheint das Badge dieser Playlist überall in dieser Farbe, mit lesbarem Text

  Scenario: Zurücksetzen auf automatische Farbe
    Given eine Playlist hat eine eigene Farbe gesetzt
    When ich sie auf der Bearbeiten-Seite zurücksetze
    Then greift wieder die automatische Farbzuordnung
```

## 3.0 How

### 3.1 Implementation Context

* **Migration**: `add_column :playlists, :color, :string` (nullable - blank/nil bedeutet
  "automatisch", analog zum bestehenden `active_library_id`-Nil-Muster).
* **`PlaylistsHelper`**:
  - `CONTEXT` wechselt von `bg-*` zu `text-bg-*` (Bootstrap-5.3-Utility, setzt Hintergrund UND
    kontrastierenden Text zusammen) - behebt den Kontrast-Bug strukturell für alle automatischen
    Farben, nicht nur `bg-light`.
  - Farbschlüssel für die Checksum wechselt von `playlist_short_name(playlist)` (voller gekürzter
    Name) zu `playlist.name.split(/\s+/).first.to_s.downcase` (erstes Wort) - macht "ähnlich
    benannt" konkret: gemeinsames erstes Wort ⇒ dieselbe Farbe. `dj_playlist?` bleibt als
    vorrangiger Sonderfall bestehen (`text-bg-dark`, unabhängig vom ersten Wort).
  - Neue Methode `playlist_badge(playlist)`: zentrales, einziges Badge-Markup
    (`content_tag(:span, playlist_short_name(playlist), class: ..., style: ...)`), ersetzt die
    fünf duplizierten `<span class="badge ...">`-Stellen. Nutzt `playlist.color` (falls vorhanden)
    als Inline-Style (`background-color` + berechnete Kontrastfarbe für den Text), sonst die
    automatische `text-bg-*`-Klasse.
  - Neue private Methode zur Kontrastberechnung (einfache Helligkeitsformel auf dem gewählten
    Hex-Wert, liefert `#000`/`#fff` für den Text) - kein zusätzliches Gem nötig.
* **`PlaylistsController#update`**: erlaubt jetzt `color` (bisher wurden gar keine Params
  verarbeitet - `@playlist.save!` speicherte nur den unveränderten Datensatz erneut). Leerer Wert
  (leerer String aus dem Formular) wird wie `nil` behandelt (`blank?`), kein Sonderfall nötig.
* **`playlists/edit.html.erb`**: `f.color_field :color` (natives `<input type="color">`,
  vorbefüllt mit der aktuellen eigenen Farbe, sonst Schwarz als Platzhalter-Startwert - auf einen
  Hex-Wert für die automatisch berechnete Farbe als Startpunkt wurde verzichtet, da das eine
  zusätzliche Bootstrap-Klasse-zu-Hex-Zuordnung gebraucht hätte, ohne echten Mehrwert gegenüber
  einfach selbst eine Farbe zu wählen) + ein `button_to "Automatische Farbe verwenden",
  playlist_path(@playlist), method: :patch, params: { playlist: { color: "" } }` zum
  Zurücksetzen.
* **Aufrufstellen**: alle fünf Stellen mit dupliziertem Badge-Markup auf `playlist_badge(playlist)`
  umgestellt (Argument bleibt `playlist` bzw. `pt.playlist`, je nach Stelle).
* **Tests**: Helper-Specs für `playlist_badge` (automatische Klasse, Override-Style, Kontrastwahl
  schwarz/weiss je nach Helligkeit, DJ-Gig-Sonderfall, gleiches erstes Wort ⇒ gleiche Farbe).
  Request-Spec für `PlaylistsController#update` (Farbe speichern, Farbe zurücksetzen). Ein
  Regressionscheck, dass `text-bg-light` (nicht mehr `bg-light` allein) irgendwo im möglichen
  Farb-Set vorkommt, plus dass keine der fünf Aufrufstellen mehr das alte, duplizierte Markup
  enthält.
