# 85 - Enhancement (Tooling): Rubocop-Konfiguration einrichten

## 1. Why

### 1.1 Objective

Dieses Projekt hat keine `.rubocop.yml` - Rubocop läuft komplett mit Cop-Defaults. Das erzeugt
einen bestehenden, nie aufgelösten Widerspruch: `CODE_GUIDELINES.md` schreibt Double-Quoted-Strings
vor, Rubocops Default für `Style/StringLiterals` ist aber `single_quotes` - das genaue Gegenteil.
Dazu kommt eine nie bereinigte Alt-Baseline von ~700 Meldungen, wodurch jede Session bisher "Delta
gegen Baseline statt 0 Offenses" prüfen musste (siehe `musicnet-workflow-erfahrungen`-Notiz) statt
sich auf eine saubere, verlässliche Basis stützen zu können. Zusätzlich sind in dieser Session
mehrfach dieselben Metrik-Schwellen (`ClassLength`, `MethodLength`, `AbcSize`) knapp überschritten
und durch Methoden-Extraktion behoben worden (`RelatedTracksFinder`, `TrackTagsController`) - ein
Hinweis, dass die Rubocop-Defaults für den in diesem Projekt üblichen Domain-Resource-Stil (Rails
Controller/Service-Objekte) eher knapp bemessen sind.

### 1.2 Context

Mit dem DJ abgestimmt: die bestehenden ~700 Meldungen werden nicht in einem grossen manuellen
Cleanup behoben, sondern per `rubocop --auto-gen-config` in eine `.rubocop_todo.yml` eingefroren -
Autokorrigierbares wird sofort gefixt, der Rest bleibt als dokumentierte Altlast bestehen, ohne
künftige Arbeit zu blockieren. Ab jetzt gilt aber: **keine neuen Verstösse ausserhalb der
Todo-Liste** - der Rubocop-Delta-Check in künftigen Sessions kann sich dann auf "0 neue Meldungen"
statt auf manuelles Baseline-Vergleichen stützen.

Konfiguration bewusst so nah wie möglich am Rubocop-Default gehalten, mit genau drei begründeten
Abweichungen (nicht mehr):
1. `Style/StringLiterals` → `double_quotes` (behebt den Widerspruch zu `CODE_GUIDELINES.md`).
2. Angehobene Metrics-Schwellen (`ClassLength`/`MethodLength`/`AbcSize`) - passend zum in dieser
   Session wiederholt beobachteten Reibungspunkt.
3. `Style/Documentation` deaktiviert - verlangt einen Doc-Kommentar über jeder Klasse, wird in
   diesem Projekt nirgends genutzt oder gewünscht.

### 1.3 Depends On Intents

Keine fachlichen Abhängigkeiten - reine Tooling-Änderung, betrifft aber potenziell jede Datei im
Repo (Autokorrektur).

## 2.0 What

```gherkin
Feature: Rubocop-Konfiguration einrichten

  Als Entwickler (Mensch oder KI-Agent) an diesem Projekt
  möchte ich eine `.rubocop.yml` mit den drei oben begründeten Abweichungen und eine
  `.rubocop_todo.yml` für bestehende Altlasten
  damit `bundle exec rubocop` künftig eine verlässliche 0-neue-Meldungen-Basis liefert statt bei
  jeder Session manuell gegen eine unbekannte Baseline verglichen werden zu müssen.

  Scenario: Konsistente String-Konvention
    Given ein neuer String ohne Interpolation wird geschrieben
    When `bundle exec rubocop` läuft
    Then wird Double-Quoting verlangt, nicht Single-Quoting

  Scenario: Bestehende Altlasten blockieren nicht
    Given eine Datei hat eine nicht-autokorrigierbare Alt-Meldung (z.B. eine zu lange Methode)
    When `bundle exec rubocop` auf dieser Datei läuft
    Then wird diese konkrete, bereits bekannte Meldung nicht erneut gemeldet (via .rubocop_todo.yml)
    But eine neue, andersartige Meldung in derselben Datei wird weiterhin gemeldet

  Scenario: Angehobene Metrik-Schwellen
    Given eine Service-Klasse mit mehreren klar benannten, je unter 15 Zeilen langen Methoden
    When ihre Gesamtlaenge die alte ClassLength-Schwelle (100) aber nicht die neue überschreitet
    Then meldet Rubocop keinen Verstoss
```

## 3.0 How

### 3.1 Implementation Context

* Keine neuen Gems (kein `rubocop-rails`/`rubocop-rspec`) - Rubocop selbst schlägt diese am Ende
  jedes Laufs vor, aber das ist ausserhalb des hier abgestimmten, bewusst minimalen Umfangs.
* `.rubocop.yml`:
  ```yaml
  require: []

  AllCops:
    NewCops: disable # verhindert die "cops not configured"-Warnung bei jedem Lauf
    TargetRubyVersion: 3.2

  Style/StringLiterals:
    EnforcedStyle: double_quotes

  Style/Documentation:
    Enabled: false

  Metrics/ClassLength:
    Max: 150

  Metrics/MethodLength:
    Max: 15

  Metrics/AbcSize:
    Max: 25

  inherit_from: .rubocop_todo.yml
  ```
  (Exakte Zahlen ggf. anhand des tatsächlichen Autokorrektur-/Auto-Gen-Ergebnisses noch verfeinert.)
* Ablauf: `bundle exec rubocop -A` (Autokorrektur inkl. String-Umstellung) → Diff-Grösse prüfen und
  stichprobenartig sichten, bevor committet wird → `bundle exec rubocop --auto-gen-config` für den
  nicht-autokorrigierbaren Rest → `.rubocop_todo.yml` prüfen (grobe Grössenordnung plausibel?) →
  volle Test-Suite laufen lassen (Autokorrektur ist rein kosmetisch, sollte nichts brechen, aber
  verifizieren) → committen.

## 4.0 Challenges & Solutions

**Herausforderung:** `CODE_GUIDELINES.md` behauptete nach der Einführung weiter, dass keine
`.rubocop.yml` existiere.

**Loesung:** Nachziehen der Dokumentation, damit Projektregeln und effektive Tool-Konfiguration wieder
konsistent sind.
