# 15 - Enhancement (Infrastruktur): Migration Phase B — Devise 4 → 5

## Why

Zweite Phase der in Intent 13 geplanten Migrations-Roadmap. Devise 4.9.3 → 5.0.4 bringt u.a. zwei
Security-Fixes (CVE-2026-32700, CVE-2026-40295) und ist Voraussetzung für Rails-8-Kompatibilität.

## Depends On Intents

* 13 - Enhancement (Infrastruktur): Migration auf aktuelle Toolchain (übergeordnete Roadmap)
* 14 - Enhancement (Infrastruktur): Migration Phase A — Ruby und Dev-Tooling

## What

Devise 4.9.3 → 5.0.4. Changelog geprüft (Breaking Changes seit 5.0.0.rc): entfernte
`:bypass`-Option, `devise_error_messages!`, `Devise::TestHelpers`, geänderte
Secret-Key-Auflösung (nutzt jetzt immer `application.secret_key_base`) — keine davon im Code
verwendet. Keine eigenen Devise-Views im Projekt, daher auch die View-Änderungen
(`<p>`-Wrapping, `data-turbo-temporary`) irrelevant.

## How

* [x] 1.0 - Changelog auf Breaking Changes prüfen:
    - [x] 1.1 - Codebase auf entfernte APIs durchsucht (`devise_error_messages!`,
          `Devise::TestHelpers`, `Devise.secret_key`, `:bypass`) — keine Treffer
    - [x] 1.2 - Eigene Devise-Views geprüft — keine vorhanden, Standard-Views der Gem greifen
* [x] 2.0 - Devise auf 5.0.4 anheben (`bundle update devise`, konfliktfrei resolved)
* [x] 3.0 - Verifikation:
    - [x] 3.1 - Testsuite (47/47 grün)
    - [x] 3.2 - Frischer Boot-Check via `bin/rails runner`
    - [x] 3.3 - Sign-in-Seite per simuliertem Request geprüft: HTTP 200, Spotify-OmniAuth-Link
          vorhanden (Devise 5 schränkt die OmniAuth-Authorize-Route auf POST-only ein — betrifft
          uns nicht, da kein eigener Login-Link im Code, nur die Standard-Devise-View)
    - [x] 3.4 - Rubocop Lint/Security (keine Funde)
