# 78 - Feature (Tags): Auftrittsdatum automatisch aus Playlist-Namen erkennen

## 1. Why

### 1.1 Objective

Viele Playlist-Namen beginnen mit einem Datum eines konkreten DJ-Auftritts (z.B.
"2023-12-01_Salsadancers": Ort "Salsadancers", Auftritt am 2023-12-01). Der DJ wollte auch dieses
Datum als Tag an den betroffenen Tracks sehen, zusätzlich zur bereits in Intent 77 gebauten
Alias-Taxonomie.

### 1.2 Context

Die Menge möglicher Daten ist nicht wie die übrige Taxonomie im Voraus aufzählbar - jeder Tag muss
dynamisch aus dem tatsächlichen Playlist-Namen erzeugt werden, nicht aus einer festen
Alias-Liste. Ein generischer Wortgrenzen-Alias-Vergleich (wie in Intent 77 für die übrige
Taxonomie gebaut) wäre hier riskant: ein kurzes Jahres-Tag wie "2026" würde per `\b2026\b`
fälschlich auch in einem präziseren Datum wie "2026-01-05..." mitmatchen, da nach der
Normalisierung (Bindestriche → Leerzeichen) beide durch Leerzeichen getrennt sind.

### 1.3 Depends On Intents

* 77 - Feature (Tags): liefert das Category/Tag/TrackTag-Datenmodell, den `assign_track_tags`-Task
  und `Tag.normalize`, die hier wiederverwendet werden.

## 2.0 What

```gherkin
Feature: Auftrittsdatum-Tags aus Playlist-Namen

  Als DJ
  möchte ich, dass ein im Playlist-Namen enthaltenes Auftrittsdatum automatisch als Tag an den
  betroffenen Tracks erscheint
  damit ich sehen kann, an welchem Termin ein Track tatsächlich gespielt wurde.

  Scenario: Volles Datum erkannt
    Given eine Playlist heisst "2023-12-01_Salsadancers"
    When der Rake-Task läuft
    Then bekommen ihre Tracks den Tag "2023-12-01" unter der Kategorie "Auftrittsdatum"

  Scenario: Unvollständiges Datum (nur Jahr-Monat oder nur Jahr)
    Given eine Playlist heisst "2021-02-Fusion" bzw. "2026 Fusionizers"
    When der Task läuft
    Then wird der Tag "2021-02" bzw. "2026" erzeugt, passend zur im Namen vorhandenen Genauigkeit

  Scenario: Kein Datum im Namen
    Given eine Playlist heisst "Blues Piano"
    When der Task läuft
    Then wird kein Auftrittsdatum-Tag für ihre Tracks erzeugt

  Scenario: Keine Verwechslung von Jahr und präziserem Datum
    Given ein Jahres-Tag "2026" existiert und eine andere Playlist heisst "2026-01-05_Anderes"
    When der Task läuft
    Then bekommen deren Tracks nur den Tag "2026-01-05", nicht zusätzlich "2026"

  Scenario: Wiederholbarkeit
    Given ein Track hatte einen Auftrittsdatum-Tag, seine Playlist wird aber umbenannt (Datum
    entfernt)
    When der Task erneut läuft
    Then verschwindet der Auftrittsdatum-Tag für diesen Track wieder
```

## 3.0 How

### 3.1 Implementation Context

* **Migration** `AddAuftrittsdatumCategory`: legt nur die Kategorie an (`is_event: true`), keine
  festen Tags - die werden dynamisch vom Task erzeugt.
* **`assign_track_tags.rake`**: Datums-Erkennung per Regex
  (`/\A(\d{4})(?:[-_](\d{2}))?(?:[-_](\d{2}))?[-_ ]/`) direkt auf dem rohen Playlist-Namen, NICHT
  über `Tag#matches_normalized_name?`/Alias-Matching - die Playlist-zu-Datum-Zuordnung ist
  dadurch deterministisch (das Regex-Ergebnis bestimmt direkt, welcher Tag gemeint ist) und umgeht
  das in 1.2 beschriebene Jahr/präziseres-Datum-Kollisionsrisiko vollständig. Tags werden für
  bereits gesehene UND neu erkannte Daten synchronisiert (`date_category.tags.reload.each`), damit
  auch ein nicht mehr vorkommendes Datum seine `TrackTag`-Zeilen wieder verliert.
  `Tag.includes(:category).to_a.reject { |tag| tag.category.name == "Auftrittsdatum" }` schliesst
  diese Tags explizit von der generischen Alias-Schleife aus, um Doppel-Verarbeitung zu vermeiden.
  Gemeinsame Synchronisierungslogik (`sync_tag`, Soll/Ist-Abgleich + Stärkeberechnung) wird mit der
  Alias-Schleife aus Intent 77 geteilt.
