# 66 - Feature (Tracks-Show): Song-Cover aus der heruntergeladenen Datei

## 1. Why

### 1.1 Objective

Die Track-Detailseite (`/tracks/:id`) zeigt bisher kein Cover-Bild. Statt dafür - wie ursprünglich
für die Artist-Alben-Tabelle diskutiert (Intent 65) - eine neue Spotify-Sync-Pipeline aufzubauen
(DB-Spalte, Backfill-Rake-Task), reicht ein einfacherer Weg: `spotdl` bettet das Cover beim
Download bereits in die `.m4a`-Datei ein, und `WahWah` (bereits für `Track#genre` im Einsatz) kann
eingebettete Bilder aus Audiodateien auslesen - ganz ohne zusätzliche Spotify-API-Aufrufe, DB-Spalte
oder Backfill.

### 1.2 Context

Bestätigt durch Lektüre des `wahwah`-Gems (`lib/wahwah/mp4_tag.rb`, `lib/wahwah/tag.rb`): das
`covr`-Atom eines MP4/`.m4a`-Containers wird als `:image`-Attribut geparst,
`WahWah::Tag#images` liefert ein Array von Hashes (`{data:, mime_type:, type: :cover}`) - erstes
Element ist das eingebettete Cover. Funktioniert nur, wenn der Track bereits heruntergeladen ist
(`Track#track_path` vorhanden); ohne Datei gibt es kein Bild.

### 1.3 Depends On Intents

* 28 - Feature (Tracks): führte `Track#genre`s Datei-Parse-Pattern (WahWah, Fehlerbehandlung) ein,
  an dem sich dieser Intent orientiert.
* 64 - Enhancement (Tracks-Show): Layout, in das das Cover eingefügt wird.
* 65 - Enhancement (Artists-Show): ursprünglicher Anstoss (Album-Cover in der Alben-Tabelle),
  daraus entstand stattdessen dieser einfachere, Song-fokussierte Ansatz.

## 2.0 What

```gherkin
Feature: Tracks-Show - Song-Cover

  Als DJ
  möchte ich auf der Track-Detailseite das Cover-Bild sehen
  damit ich den Song auch visuell wiedererkenne.

  Scenario: Cover eines heruntergeladenen Tracks mit eingebettetem Bild
    Given ein Track wurde heruntergeladen und seine Datei enthält ein eingebettetes Cover
    When ich seine Detailseite besuche
    Then sehe ich das Cover-Bild neben dem Titel

  Scenario: Kein Cover ohne heruntergeladene Datei
    Given ein Track wurde noch nicht heruntergeladen
    When ich seine Detailseite besuche
    Then sehe ich kein Cover-Bild (auch keinen kaputten Bild-Platzhalter)

  Scenario: Kein Cover, wenn die Datei keins eingebettet hat
    Given ein Track wurde heruntergeladen, seine Datei enthält aber kein eingebettetes Cover
    When ich seine Detailseite besuche
    Then sehe ich kein Cover-Bild (auch keinen kaputten Bild-Platzhalter)
```

## 3.0 How

### 3.1 Implementation Context

* **`Track#cover_image`** (neue Methode, privat genutzte Parse-Logik analog zu
  `#read_genre_from_file`): `return nil unless track_path`, dann
  `WahWah.open(track_path).images.first` (liefert `nil`, wenn das Array leer ist), rescued gegen
  dieselben Fehlerklassen wie `#genre` (`WahWah::WahWahArgumentError`,
  `WahWah::WahWahNotImplementedError`). Kein DB-Cache wie bei `#genre` - das lohnt sich nur bei
  Aufrufen über viele Tracks hinweg (Tabellen-Rendering), hier wird nur ein einzelner Track auf
  seiner eigenen Seite einmal pro Bildanfrage geparst.
* **Neue Route** `GET /tracks/:id/cover` (member route auf `resources :tracks`) →
  `TracksController#cover`: `image = Track.find(params[:id]).cover_image`, `head :not_found`
  wenn `nil`, sonst `send_data image[:data], type: image[:mime_type], disposition: "inline"`.
* **View** (`tracks/show.html.erb`): `<%= image_tag cover_track_path(@track), ... %>` nur wenn
  `@track.track_path.present?` (dieselbe Bedingung wie beim Spotify-Fallback, Intent 64) - mit
  `onerror="this.remove()"`, damit eine Datei ohne eingebettetes Cover (Bild-Request liefert 404)
  keinen kaputten Bild-Platzhalter hinterlässt, sondern das `<img>` clientseitig entfernt wird.
  Platziert links neben dem Titel im bestehenden `d-flex`-Kopfbereich.
* **Tests**: Model-Spec für `Track#cover_image` (mit `track_path`, ohne, mit einer Datei ohne
  Bild-Atom); Request-Specs für `GET /tracks/:id/cover` (200 mit Content-Type, 404 ohne Datei/ohne
  Bild) und für `tracks#show` (Cover-`<img>` vorhanden/fehlt je nach `track_path`). Für die
  Model-/Request-Specs wird eine kleine Test-Fixture-`.m4a`-Datei mit eingebettetem Cover gebraucht
  (bestehende `spec/fixtures/files/silence.m4a` hat keins) - per `ffmpeg` oder einer bereits
  gepflegten Fixture erzeugt, falls im Projekt vorhanden, sonst neu erzeugt und eingecheckt.

## 4.0 Challenges & Solutions

**Herausforderung:** `spec/requests/tracks_spec.rb` lag bereits nahe an der bestehenden
`Metrics/BlockLength`-Baseline; die neuen Cover-Beispiele hätten einen frischen Rubocop-Verstoss
erzeugt.

**Loesung:** Cover-spezifische Request-Beispiele in eine eigene Datei `spec/requests/track_cover_spec.rb`
ausgelagert, statt die Todo-Baseline zu erweitern.
