# 89 - Feature (Playlists-Tracks-Tags): Tag-Zuweisung und Entfernen vereinheitlichen

## 1. Why

Der DJ möchte Tracks auch direkt in der Playlist-Ansicht taggen können - genau gleich wie auf
`/tracks`. Ausserdem fehlt auf `/tracks` selbst (und damit auch künftig in Playlists) eine
Möglichkeit, ein zugewiesenes Tag wieder zu entfernen - das geht heute nur auf der
Track-Detailseite.

### 1.1 Context

Tags werden aktuell an drei Stellen unterschiedlich dargestellt:

* `/tracks` (Index) und `/artists/:id` (gemeinsame Partial `tracks/_tracks.erb` ->
  `tracks/_track.erb` -> `tracks/_tag_cell.erb`): interaktive Badges mit Stärke-Bearbeitung
  (`components/_track_tag_badge.erb`, Intent 81) + "+"-Widget zum Zuweisen eines bestehenden Tags
  (`tracks/_tag_assign_inline.erb`, Intent 83) - aber ohne Entfernen-Möglichkeit.
* `/playlists/:id` (eigene Partial `playlist_tracks/_playlist_track.erb`): nur statische Badges
  über den `track_tag_badge`-Helper, kein Zuweisen, kein Entfernen.
* `/tracks/:id` (Detailseite, eigene Partial `tracks/_tag_panel.html.erb`, Intent 87 Bugfix): volles
  Zuweisen-Formular (inkl. neuer Tags mit Kategorie) UND ein "×"-Entfernen-Button pro Tag.

`TrackTagsController#destroy` existiert bereits und funktioniert auf der Detailseite - antwortet
dort aber nur mit einem klassischen Redirect (`format.html`), keinem Turbo-Stream. Ein "×"-Button
in einer Liste (`/tracks`, Playlists) würde damit bei jedem Entfernen die ganze Seite zur
Track-Detailseite wegnavigieren - unerwünscht, analog zum bereits behobenen `data-turbo="false"`-
Bug bei der Zuweisung (Intent 87 Bugfix).

`ArtistsController#show` fehlte kürzlich `@recent_tag_suggestions`, obwohl die wiederverwendete
Tag-Cell-Partial das braucht (Bugfix, gleicher Tag wie dieses Intent). `PlaylistsController#show`
hat dasselbe Problem noch nicht behoben, weil die Playlist-Ansicht die Tag-Cell-Partial bisher gar
nicht verwendet.

## 2.0 What

```gherkin
Feature: Tags in Playlists zuweisen und ueberall entfernen koennen

  Als DJ
  moechte ich Tracks direkt in der Playlist-Ansicht taggen und ein Tag von jeder Ansicht aus
  wieder entfernen koennen
  damit ich nicht extra auf die Track-Detailseite wechseln muss.

  Scenario: Tag in der Playlist-Ansicht zuweisen
    Given ich bin auf der Detailseite einer Playlist
    When ich einem Track dort ein bestehendes Tag zuweise
    Then wird das Tag gespeichert und erscheint sofort als Badge in dieser Zeile

  Scenario: Tag von der Tracks-Liste aus entfernen
    Given ein Track auf /tracks hat ein zugewiesenes Tag
    When ich das Tag ueber den Entfernen-Button in der Zeile entferne
    Then verschwindet das Tag aus der Zeile, ohne dass die Seite die Liste verlaesst

  Scenario: Tag von der Playlist-Ansicht aus entfernen
    Given ein Track in einer Playlist hat ein zugewiesenes Tag
    When ich das Tag ueber den Entfernen-Button entferne
    Then verschwindet das Tag, ohne dass die Seite die Playlist-Ansicht verlaesst

  Scenario: Tag von der Track-Detailseite aus entfernen (bestehendes Verhalten bleibt)
    Given ich bin auf der Detailseite eines Tracks
    When ich ein Tag dort entferne
    Then verschwindet es weiterhin wie bisher
```

## 3.0 How

### 3.1 Implementation Context

* **Playlists auf dieselbe Tag-Cell-Partial umstellen:** `playlist_tracks/_playlist_track.erb`
  ersetzt seinen bisherigen statischen `track_tag_badge`-Loop durch
  `render partial: "tracks/tag_cell", locals: { track: track, recent_tags: @recent_tag_suggestions }`
  - identischer Aufruf wie in `tracks/_track.erb`. `PlaylistsController#show` ergaenzt
  `@recent_tag_suggestions` (gleicher Aufruf wie in `TracksController#index`/`ArtistsController#show`).
* **Entfernen-Button in der gemeinsamen Badge-Komponente:** `components/_track_tag_badge.erb`
  bekommt einen kompakten "×"-Button (`button_to`, `method: :delete`, `turbo_confirm`) - dadurch
  automatisch ueberall verfuegbar, wo diese Partial gerendert wird (`/tracks`, `/artists/:id`,
  `/playlists/:id`), ohne die Partial dreifach anzupassen.
* **`TrackTagsController#destroy` turbo-stream-faehig machen:** analog zu `#create` (Intent 87
  Bugfix) - `respond_to`-Block mit `format.turbo_stream` (ersetzt `dom_id(track, :track_tags_cell)`
  UND `dom_id(track, :track_tags_panel)`, ein `turbo_stream.replace` auf ein nicht vorhandenes Ziel
  ist ein stiller No-Op) statt ausschliesslich `format.html { redirect_to ... }`. Die Detailseite
  behaelt ihr bisheriges Verhalten (ihr eigener "×"-Button in `_tag_panel.html.erb` bleibt
  bestehen, profitiert aber zusaetzlich vom Turbo-Stream statt eines vollen Redirects).
* **Kein neuer Endpunkt noetig** - `track_tag_path(tt)`/`TrackTagsController#destroy` existiert
  bereits, wird nur um den Turbo-Stream-Zweig erweitert.

## 4. Challenges & Solutions

Der neue "×"-Button liess bei laengeren Tag-Namen den `.col-tags`-Zelleninhalt sichtbar in die
Nachbarspalte (Datei-Buttons) hineinlaufen - `table-layout: fixed` begrenzt nur die Zellbreite,
nicht zwangsweise ungewrappte Inline-Flex-Inhalte darin. Ein System-Spec (Cuprite) schlug darum mit
`MouseEventFailed` fehl (Klick traf den ueberlappenden Datei-Button statt des "×"). Gleiche Lehre
wie beim 🎧-Button (Intent 51 Nachtrag): `flex-wrap` auf den Badge+Button-Wrapper ergaenzt, damit
der "×"-Button bei Bedarf auf eine eigene Zeile faellt statt zu ueberlappen. Zusaetzlicher
Layout-Regressionstest in `spec/system/track_row_buttons_layout_spec.rb` (Bounding-Box-Vergleich),
damit dieser DOM-vorhanden-aber-visuell-ueberlappt-Fehlermodus kuenftig automatisch auffaellt.
