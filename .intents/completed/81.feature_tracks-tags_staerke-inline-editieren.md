# 81 - Feature (Tracks-Tags): Stärke inline in der Tabelle editieren

## 1. Why

### 1.1 Objective

Die manuelle Tag-Zuweisung (Intent 79) setzt die Stärke nur beim erstmaligen Zuweisen auf der
Track-Detailseite. Beim täglichen Gebrauch stellte sich heraus, dass das Feintuning einer bereits
gesetzten Stärke mühsam ist (Tag entfernen, neu zuweisen, Stärke neu eingeben) - der DJ wollte die
Stärke direkt in der Tracks-Übersicht (`/tracks`) korrigieren können, ohne auf die Detailseite zu
wechseln: Klick auf das Tag-Badge, Wert am Zahlenfeld ("Rädchen") anpassen, Enter speichert.

### 1.2 Context

Nachträglich (nach der ersten Umsetzung) kamen drei Bedienungs-Korrekturen aus der Praxis dazu, in
diesem Intent mit erledigt statt als eigener Intent, da sie direkte Verfeinerungen desselben
Interaktionsmusters sind, keine neue Funktion:

* Ein Klick ausserhalb eines gerade geöffneten Editierfelds soll es ohne Speichern schliessen.
* Das Anklicken eines anderen Tags soll das zuvor geöffnete automatisch schliessen.
* Erfolgreiches Speichern soll als kurzes visuelles Aufglimmen sichtbar sein.

### 1.3 Depends On Intents

* 77 - Feature (Tags): liefert `Category`/`Tag`/`TrackTag` inkl. `strength`-Spalte.
* 79 - Feature (Tracks): manuelle Tag-Zuweisung inkl. `track_tag_badge`-Helper, dessen Anzeige
  hier wiederverwendet und um Inline-Editierbarkeit ergänzt wird.

## 2.0 What

```gherkin
Feature: Tag-Stärke inline in der Tracks-Tabelle editieren

  Als DJ
  möchte ich die Stärke eines bereits zugewiesenen Tags direkt in der Tracks-Übersicht anpassen
  damit ich das Feintuning nicht über die Track-Detailseite umständlich per Entfernen+Neuzuweisen
  machen muss.

  Scenario: Stärke per Klick, Zahlenfeld und Enter ändern
    Given ein Track mit einem zugewiesenen Tag wird in der Tracks-Tabelle angezeigt
    When ich auf das Tag-Badge klicke, den Wert im Zahlenfeld ändere und Enter drücke
    Then wird die neue Stärke gespeichert und sofort im Badge angezeigt

  Scenario: Ungültige Stärke wird abgelehnt
    Given das Editierfeld für ein Tag ist geöffnet
    When ich einen Wert ausserhalb 1-10 eingebe und speichere
    Then bleibt die alte Stärke bestehen und das Formular zeigt die Fehlermeldung erneut an

  Scenario: Abbruch per Escape
    Given das Editierfeld für ein Tag ist geöffnet
    When ich Escape drücke
    Then schliesst sich das Feld ohne zu speichern und die alte Stärke bleibt bestehen

  Scenario: Abbruch per Klick ausserhalb
    Given das Editierfeld für ein Tag ist geöffnet
    When ich irgendwo ausserhalb des Tags klicke
    Then schliesst sich das Feld ohne zu speichern

  Scenario: Ein anderes Tag anklicken schliesst das vorherige
    Given das Editierfeld für Tag A ist geöffnet
    When ich auf das Badge von Tag B klicke
    Then schliesst sich Tag A's Editierfeld und Tag B's Editierfeld öffnet sich

  Scenario: Erfolgsfeedback
    Given ich habe eine neue Stärke gespeichert
    Then glimmt das Badge kurz auf, um den erfolgreichen Speichervorgang sichtbar zu machen
```

## 3.0 How

### 3.1 Implementation Context

* **Route/Controller**: `resources :track_tags` um `:update` ergänzt (`config/routes.rb`);
  `TrackTagsController#update` liest die Stärke sowohl genestet (`track_tag[strength]`, vom
  `form_with model:`-Formular) als auch flach (`params[:strength]`, für einfache Request-Specs)
  und setzt `@saved`/`@track_tag` für das Turbo-Stream-Template.
* **View**: neues Partial `components/_track_tag_badge.erb` kapselt Anzeige- und Editieransicht in
  einem Wrapper mit stabiler `dom_id(track_tag, :tag_badge)`; `tracks/_track.erb` rendert es statt
  direkt `track_tag_badge(tt)`. `track_tags/update.turbo_stream.erb` ersetzt den Wrapper nach dem
  Speichern - bei Erfolg mit der Badge-Ansicht (+ `just_saved: true` fürs Aufglimmen), bei einem
  Validierungsfehler erneut mit dem Formular (`editing: true`) inkl. Fehlermeldung.
* **Stimulus (`tag_strength_controller.js`)**: `edit()`/`cancel()` blenden Badge/Formular um;
  `outsideClick()`, per `click@window`-Action auf dem Wrapper registriert, deckt sowohl "Klick
  ausserhalb" als auch "Klick auf ein anderes Tag" mit derselben Prüfung ab (Klick-Ziel liegt
  nicht innerhalb des eigenen Elements + Formular ist gerade offen). Der eigentliche Speichervorgang
  ist ein normaler Turbo-Formular-Submit auf Enter, keine eigene Fetch-Logik nötig.
  `tracks/show.html.erb` bleibt bewusst unverändert - das dortige Badge ist ein Link auf die
  gefilterte Trackliste, ein Klick-zum-Editieren würde damit kollidieren.
* **Feedback**: CSS-Keyframe-Animation `tag-glow` (`application.scss`) statt einer Transition, da
  sie nach dem Turbo-Replace einmalig automatisch abläuft, ohne dass JS die Klasse wieder entfernen
  muss.

## Challenges & Solutions

* Der erste Versuch scheiterte in der System-Spec (nicht im Request-Spec): `form_with model:
  track_tag` sendet die Stärke genestet als `track_tag[strength]`, während der Controller zunächst
  nur das flache `params[:strength]` (Muster von `#create`s `number_field_tag`) las. Erst der
  echte Browsertest (Cuprite) deckte das auf, der Request-Spec hatte versehentlich flache Parameter
  simuliert - Beispiel dafür, warum `AGENTS.md`/`CLAUDE.md` echte Browser-Verifikation bei
  Turbo/Stimulus-Interaktionen verlangen.
