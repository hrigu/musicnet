# 23 - Enhancement (Spotify-Playlists): Sync-Transaktionen, Parallelitäts-Schutz und Log-Pegel

## 1. Why

### 1.1 Objective

Der Sync (voll und einzeln) soll pro Playlist eine Transaktion nutzen statt eine pro Datensatz,
parallele Sync-Läufe sollen sauber abgewiesen werden, und die Sync-Logs sollen im Normalbetrieb
nicht mehr kilometerlang sein.

### 1.2 Context

Ein Log-Auszug eines vollen Syncs zeigte: (1) Hunderte Mini-Transaktionen (BEGIN/COMMIT pro
`find_or_create_by!`), (2) zwei parallel laufende Syncs, die sich den SQLite-Write-Lock streitig
machten (`BEGIN immediate TRANSACTION (511.2ms)` = halbe Sekunde Lock-Wartezeit), (3) eine
Info-Logzeile pro Track/Album/Artist. SQLite erlaubt nur einen Schreiber — parallele Syncs
bremsen sich massiv gegenseitig und riskieren `SQLite3::BusyException`.

## 2.0 What

```gherkin
Feature: Spotify-Playlists - Sync-Transaktionen, Parallelitäts-Schutz und Log-Pegel

  Als DJ
  möchte ich dass der Sync pro Playlist atomar schreibt, sich nicht parallel starten lässt
  und leise loggt
  damit der Sync schnell läuft, die DB konsistent bleibt und das Log lesbar ist.

  Scenario: Playlist-Import ist atomar
    Given der Import eines Tracks einer Playlist schlägt fehl
    When der Sync diese Playlist verarbeitet
    Then wird keine halb importierte Playlist zurückgelassen (Rollback)

  Scenario: Paralleler Sync wird abgewiesen
    Given ein Sync läuft bereits
    When ein zweiter Sync (voll oder Einzel-Refresh) gestartet wird
    Then wird er mit einer verständlichen Meldung abgewiesen
    And der Benutzer sieht die Meldung als Alert

  Scenario: Leises Log
    Given der Sync verarbeitet Tracks, Alben und Artists
    When das Log-Level auf info steht
    Then erscheint pro Playlist eine Zeile, aber keine pro Track/Album/Artist
```

## 3.0 How

### 3.1 Implementation Context

Alle Eingriffe in `BuildMusicNetService` plus kleine Controller-Anpassungen:

1. **Transaktion pro Playlist:** Die DB-Schreibphase von `build_playlist` und
   `refresh_playlist` wird in `ActiveRecord::Base.transaction(requires_new: true)` gehüllt.
   `requires_new: true`, damit auch unter der Fixture-Transaktion der Tests (und in allfälligen
   künftigen umgebenden Transaktionen) ein echter Savepoint entsteht und Atomarität gilt.
   Trade-off: Die Transaktion bleibt während der Lazy-API-Calls von rspotify
   (popularity/audio_features neuer Tracks) offen — akzeptabel, da der Parallelitäts-Schutz
   (Punkt 2) konkurrierende Schreiber ausschliesst und Lese-Requests unter SQLite/WAL nicht
   blockiert werden.
2. **Parallelitäts-Schutz:** Klassenweiter `Mutex` (`SYNC_LOCK`) mit `try_lock`; läuft bereits
   ein Sync (voll oder Refresh), wird `SyncAlreadyRunningError` geworfen.
   `PlaylistsController#fetch_all`/`#refresh` fangen den Fehler und zeigen einen Alert.
   Ein Prozess-Mutex genügt: Die App läuft als einzelner Puma-Prozess (Single-User-App).
3. **Log-Pegel:** Die Logzeilen pro Track/Album/Artist (`build_track:`, `build_album:`,
   `build_artists:`) wechseln von `info` auf `debug`; die Zeile pro Playlist bleibt `info`.
   Kein Test — reines Logging ohne Verhaltensänderung.
