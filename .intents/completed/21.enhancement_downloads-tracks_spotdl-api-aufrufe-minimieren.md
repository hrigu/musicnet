# 21 - Enhancement (Downloads-Tracks): spotdl-API-Aufrufe minimieren

## 1. Why

### 1.1 Objective

Der Nachlade-Weg für fehlende Audiodateien (`tracks#download` → `DownloadTrackService`) soll
deutlich weniger Spotify-API-Requests auslösen, damit spotdl nicht mehr ins Rate-Limit läuft
(zuletzt: 24-Stunden-Sperre nach einem Batch mit ~37 Track-URLs).

### 1.2 Context

`DownloadTrackService` übergibt spotdl einzelne Track-URLs (`spotdl download <url> <url> …`).
spotdl löst jede URL separat über die Spotify-API auf (Track-, Album-, Artist-Requests).
`DownloadPlaylistService` nutzt dagegen `spotdl sync <playlist-url> --save-file <name>.spotdl`:
Die Playlist wird gebündelt geholt (100 Tracks pro Request) und die Song-Metadaten werden in
der `.spotdl`-Datei gecached — bereits vorhandene Dateien überspringt spotdl.

Fehlende Tracks hängen (nach dem Orphan-Cleanup des Syncs) immer an mindestens einer Playlist.
Statt Track für Track aufzulösen, kann der Nachlade-Weg also pro betroffener Playlist einen
`spotdl sync` fahren.

**Vorsicht als Voraussetzung:** `spotdl sync` löscht standardmässig Dateien von Songs, die
nicht mehr in der Playlist sind. `DownloadPlaylistService#build_command` definiert die Option
`--sync-without-deleting` zwar, hängt sie aber nicht an den Befehl an (toter Code). Da alle
Playlists denselben `downloads/tracks`-Ordner teilen, könnte ein Sync sonst Dateien löschen,
die eine andere Playlist noch braucht. Dieser Bug muss zuerst gefixt werden.

## 2.0 What

```gherkin
Feature: Downloads Tracks - spotdl-API-Aufrufe minimieren

  Als DJ
  möchte ich fehlende Audiodateien playlist-weise per spotdl sync nachladen
  damit spotdl die Spotify-API nur gebündelt abfragt und nicht ins Rate-Limit läuft.

  Scenario: Sync löscht keine Dateien mehr
    Given eine Playlist wird mit spotdl synchronisiert
    When ein Song auf Spotify aus der Playlist entfernt wurde
    Then bleibt seine bereits heruntergeladene Datei erhalten

  Scenario: Fehlende Tracks werden playlist-weise nachgeladen
    Given mehrere Tracks ohne Audiodatei, verteilt auf zwei Playlists
    When ich den Track-Download starte
    Then wird pro betroffener Playlist genau ein spotdl-sync ausgeführt
    And es werden keine einzelnen Track-URLs an spotdl übergeben

  Scenario: Playlist ohne fehlende Tracks
    Given alle Tracks einer Playlist haben bereits eine Audiodatei
    When ich den Track-Download starte
    Then wird für diese Playlist kein spotdl-sync ausgeführt
```

## 3.0 How

### 3.1 Implementation Context

Zwei Eingriffe, beide TDD (Red-Green-Refactor) mit den bestehenden Spec-Mustern
(`system` wird gestubbt — spotdl ist die externe Grenze):

1. `DownloadPlaylistService#build_command`: das bereits definierte `--sync-without-deleting`
   tatsächlich in den Befehl aufnehmen; bestehende Command-Erwartungen in
   `download_playlist_service_spec.rb` anpassen.
2. `DownloadTrackService#download`: statt `spotdl download` mit Track-URLs die betroffenen
   Playlists der übergebenen Tracks ermitteln (`tracks.flat_map(&:playlists).uniq`) und pro
   Playlist `DownloadPlaylistService#download` ausführen. Das öffentliche Interface
   (`new(tracks)` / `#download`) und `TracksController#download` bleiben unverändert.
