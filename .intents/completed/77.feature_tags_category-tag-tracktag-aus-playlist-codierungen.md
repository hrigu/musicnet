# 77 - Feature (Tags): Category/Tag/TrackTag-Datenmodell aus Playlist-Codierungen

## 1. Why

### 1.1 Objective

Playlist-Namen dieses DJs codieren weit mehr als nur Genre - z.B. "Blues mit Violine", "Fusion
sad", "Blues für Paula" tragen Informationen zu Instrumentierung, Emotion, Widmung usw., die
bisher nirgends strukturiert erfasst waren. Nur die Bibliothek (Blues/Fusion/Lindy-Hop) war eine
echte DB-Beziehung (`Library`). Ziel: diese Codierung in ein echtes Datenmodell überführen -
`Category` → `Tag` → `TrackTag` (Join mit `strength` 1-10, wie stark ein Tag auf einen bestimmten
Track zutrifft) - und Tracks per Rake-Task aus den aktuellen Playlist-Namen automatisch
verschlagworten, inklusive Verwaltungs-UI, Anzeige und Suche.

### 1.2 Context

Eine Analyse aller 247 Playlist-Namen (siehe Chat-Verlauf) ergab eine Taxonomie aus 12 Kategorien
mit insgesamt ca. 142 Tags (Instrumentierung, Emotion/Stimmung, Qualität/Level, Tempo/Energie,
Geografie/Kultur, Musikstil/Genre, Person/Widmung, Funktion/Setzweck, Bildhaft/Motiv,
Textur/Klangcharakter, Thema/Gesellschaft, Anlass/Ort).

Geklärte Entscheidungen (Rückfragen an den DJ):
* **Stärke**: häufigkeitsbasiert - je öfter ein Track in Playlists vorkommt, die auf denselben Tag
  matchen, desto höher die Stärke (1×→5, 2×→7, 3+×→10).
* **Ausführung**: nur als wiederholbarer Rake-Task, NICHT in den Spotify-Sync
  (`BuildMusicNetService`) integriert - die Taxonomie wird laufend im Admin-UI angepasst, der Task
  muss darum beliebig oft neu laufen können.
* **Verwaltung**: eigene Admin-UI wie bei Bibliotheken, aber als eine kombinierte Seite statt
  zweier getrennter CRUD-Ressourcen (bei ~12 Kategorien mit je einer Handvoll Tags passender als
  Libraries' zwei-Index-Seiten-Muster).
* **Umfang**: inkl. Anzeige (Tags-Spalte/Badges) und Suche (`tag:`-Feld), `tag_strength:` bewusst
  nicht gebaut (siehe 3.1).

Kritischer, real aufgetretener Bug während der Analyse: reiner Teilstring-Vergleich (wie
`Library#matching`) hätte den Alias "sad" fälschlich auch in "Salsadancers" gematcht - das
Tag-Matching braucht darum Wortgrenzen (`\b...\b`) statt `include?`.

### 1.3 Depends On Intents

* 57 - Feature (Bibliotheken): `Library`/`LibraryPlaylist` sind das direkte Vorbild für das
  Category/Tag-Datenmodell und die Seed-Migrations-Konvention (lokale, tabellen-gebundene
  Migrations-Modelle).
* 43 - Feature (Tracks-Suche): DSL-Suche (`FIELD_SCOPES`/`TrackQueryParser`), in die `tag:` sich
  ohne Parser-Änderung einfügt.
* 71 - Feature (Playlists): Farbkonzept-Vorbild für die spätere, in Intent 79/Folgearbeit
  verfeinerte Tag-Badge-Farbe.

## 2.0 What

```gherkin
Feature: Playlist-Codierungen als Tags/Kategorien mit Stärke

  Als DJ
  möchte ich, dass die in Playlist-Namen versteckte Information (Emotion, Instrumentierung, Genre,
  Widmung usw.) automatisch als strukturierte Tags an meinen Tracks landet, mit einer Stärke, wie
  sehr der Tag zutrifft, verwaltbar über eine eigene Oberfläche und durchsuchbar
  damit ich Tracks nach mehr als nur Genre/Bibliothek filtern und wiederfinden kann.

  Scenario: Automatische Verschlagwortung nach Playlist-Namen
    Given ein Track kommt in einer Playlist "Blues mit Violine" vor
    When der Rake-Task assign_track_tags läuft
    Then bekommt der Track den Tag "Violine" unter der Kategorie "Instrumentierung"

  Scenario: Häufigkeitsbasierte Stärke
    Given ein Track kommt in zwei verschiedenen Playlists vor, die beide auf denselben Tag matchen
    When der Task läuft
    Then ist die Stärke des Tags für diesen Track 7 statt 5 (einmaliges Vorkommen)

  Scenario: Keine falschen Treffer durch Teilstrings
    Given eine Playlist heisst "Fusion Salsadancers" und ein Tag hat den Alias "sad"
    When der Task läuft
    Then bekommt kein Track dieser Playlist den Tag "Traurig" (Wortgrenzen-Check)

  Scenario: Wiederholbarkeit / Selbstheilung
    Given ein Tag wurde bereits einem Track zugeordnet, die Playlist wird aber umbenannt oder der
    Track daraus entfernt
    When der Task erneut läuft
    Then verschwindet die veraltete Zuordnung wieder

  Scenario: Verwaltung im Admin-UI
    Given ich bin auf /categories
    Then sehe ich alle Kategorien mit ihren Tags und kann Kategorien/Tags anlegen, bearbeiten,
    löschen

  Scenario: Suche nach Tag
    Given ein Track hat den Tag "Traurig"
    When ich auf /tracks nach "tag:traurig" suche
    Then finde ich diesen Track
```

## 3.0 How

### 3.1 Implementation Context

* **Migrationen**: `categories` (`name` unique, `color` nullable, `is_event` boolean),
  `tags` (`category_id` FK, `name`, `aliases` text, unique `[category_id, name]`), `track_tags`
  (`track_id`/`tag_id` FK, `strength` integer, unique `[track_id, tag_id]`). Seed-Migration
  `SeedCategoriesAndTags` mit lokalen, tabellen-gebundenen Modellklassen (Muster aus
  `SeedLibrariesAndAssignExistingPlaylists`, Intent 57) und `find_or_create_by!` - läuft die
  Migration versehentlich erneut, bleiben im Admin-UI bereits bearbeitete Zeilen unangetastet.
* **`Tag#matches_normalized_name?`/`.normalize`**: Wortgrenzen statt Teilstring-Vergleich (siehe
  1.2), inkl. Normalisierung von Apostrophen/Bindestrichen/Unterstrichen/Schrägstrichen zu
  Leerzeichen (nicht Entfernen - sonst verschmilzt z.B. "rock'n'roll" zu einem Wort und verliert
  seine Wortgrenze).
* **`lib/tasks/assign_track_tags.rake`**: berechnet pro Tag den vollen Soll-Zustand aus den
  aktuellen Playlist-Namen neu (nicht nur additiv) und gleicht ihn mit dem Ist-Zustand ab (Upsert +
  Löschen überzähliger `TrackTag`-Zeilen) - garantiert Wiederholbarkeit nach Taxonomie-Änderungen.
* **Admin-UI**: `CategoriesController`/`TagsController`, `resources :categories do resources
  :tags, shallow: true end`, eine kombinierte `categories#index`-Seite statt zweier CRUD-Seiten.
* **Suche**: `Track::FIELD_SCOPES["tag"] = :by_tag`, `TEXT_FIELDS` erweitert, `by_tag` nach dem
  Subquery-Muster von `by_playlist` (Schnittmenge bei wiederholtem `tag:`-Token).
  `tag_strength:` bewusst NICHT gebaut: zwei unabhängige Tokens (`tag:x tag_strength:>5`) könnten
  in diesem DSL-Design nicht ausdrücken "dieser bestimmte Tag hat diese Stärke", nur "hat
  irgendeinen Tag mit dieser Stärke" - ein stiller Korrektheits-Fehler. In
  `doc/track_search_syntax.md` als bekannte Lücke dokumentiert statt stillschweigend ausgelassen.
* **Anzeige**: `track_tag_badge`/`tag_badge` in `TracksHelper`, Tags-Spalte auf `/tracks`,
  Playlist-Detailansicht und Track-Detailseite (dort nach Kategorie gruppiert).
* **CSS**: `.table-tracks-detailed` von 12 auf 13 `nth-child`-Spalten erweitert, "Datei"-Spalte
  bewusst bei 9% belassen (siehe Intent-51-Regression).
