# 63 - Bug (Artists-Show): Sortierung navigiert zu /tracks statt in der Seite zu sortieren

## 1. Why

### 1.1 Objective

Auf der Künstler-Detailseite (`/artists/:id`) will der DJ die dort gezeigte Track-Tabelle
sortieren können, z.B. nach Dauer oder Bekanntheit. Klickt man auf eine Sortier-Spaltenüberschrift,
landet man stattdessen auf `/tracks` — die eigentliche Künstlerseite geht verloren.

### 1.2 Context

Root Cause: `artists/show.html.erb` rendert dasselbe geteilte Partial `tracks/_tracks.erb` wie
`tracks#index` (Sortier-Header inklusive). Dessen Sortier-Links werden von
`TracksHelper#sort_link` erzeugt, das den Ziel-Pfad fest auf `tracks_path(query)` verdrahtet —
unabhängig davon, von welcher Seite aus das Partial gerendert wird. Selbst wenn man auf der
Artist-Seite bliebe, würde aktuell trotzdem nicht sortiert: `ArtistsController#show`/
`Artist.for_show` wenden `sort`/`direction`-Parameter gar nicht erst an (im Gegensatz zu
`TracksController#index`, das dafür bereits `Track.sorted` mit der bestehenden
`Track::SORT_COLUMNS`-Whitelist nutzt).

### 1.3 Depends On Intents

* 34 - Feature (Tracks-Index): Paginierung, Sortierung, Volltextsuche — führte `sort_link` und
  `Track::SORT_COLUMNS`/`Track.sorted` ein.

## 2.0 What

```gherkin
Feature: Artist-Detailseite - Tracks in der Seite sortieren

  Als DJ
  möchte ich die Track-Tabelle auf der Künstler-Detailseite sortieren können
  ohne dabei die Künstlerseite zu verlassen.

  Scenario: Sortier-Link auf der Artist-Seite bleibt auf der Artist-Seite
    Given ich bin auf der Detailseite eines Künstlers
    When ich auf eine Sortier-Spaltenüberschrift klicke
    Then bleibe ich auf der Detailseite dieses Künstlers
    And die Tracks-Tabelle ist nach der gewählten Spalte sortiert

  Scenario: Sortierung auf /tracks bleibt unverändert
    Given ich bin auf /tracks
    When ich auf eine Sortier-Spaltenüberschrift klicke
    Then werde ich weiterhin innerhalb von /tracks sortiert, wie bisher
```

## 3.0 How

### 3.1 Implementation Context

* **`TracksHelper#sort_link`**: `tracks_path(query)` wird durch `url_for(query)` ersetzt.
  `url_for` ergänzt in einem View-Kontext fehlende Angaben (Controller/Action sowie Pfad-Parameter
  wie die Artist-`:id`) automatisch aus dem aktuellen Request — dadurch bleibt der erzeugte Link
  automatisch auf der jeweils aktuellen Seite (`/tracks` bleibt `/tracks`, `/artists/:id` bleibt
  `/artists/:id`), ohne dass `sort_link` selbst wissen muss, von welcher Seite aus es aufgerufen
  wird.
* **`ArtistsController#show`**: `@tracks` wird zusätzlich durch
  `.sorted(params[:sort], params[:direction])` geschickt — derselbe Aufruf, den
  `TracksController#index` bereits nutzt. `Artist.for_show(artist)` liefert weiterhin eine
  ActiveRecord-Relation (`artist.tracks.preload(...).strict_loading`), auf die sich `sorted`
  genauso verkettet lässt.
* Auf der Artist-Seite gibt es (anders als `/tracks`) keinen `turbo_frame_tag` um die Tabelle -
  ein Sortier-Klick dort bleibt ein normaler, voller Turbo-Drive-Seitenaufruf. Das ist für diesen
  Bugfix ausreichend; ein Turbo-Frame-Update ohne vollen Reload ist nicht Teil dieses Intents.
* **Tests:** Request-Spec auf `artists#show`, die verifiziert, dass `sort`/`direction`-Parameter
  die Reihenfolge der gerenderten Tracks ändern (analog zu bestehenden Sortier-Tests für
  `tracks#index`). Zusätzlich ein System-Spec, das einen echten Klick auf einen Sortier-Header auf
  der Artist-Seite simuliert und prüft, dass die URL weiterhin die Artist-ID enthält (nicht zu
  `/tracks` wechselt) und sich die Zeilenreihenfolge ändert — reines Link-/Navigationsverhalten,
  in einer Request-Spec nicht beobachtbar. Bestehende Sortier-Tests für `/tracks` müssen
  unverändert grün bleiben (Regressionsschutz für den geänderten `sort_link`).
