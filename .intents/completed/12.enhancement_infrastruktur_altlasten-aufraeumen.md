# 12 - Enhancement (Infrastruktur): Altlasten aufräumen vor der Migration

## Why

Bevor auf Rails 8 migriert wird (siehe Intent 13), soll bekannter toter/riskanter Code entfernt
werden. Das macht den Migrations-Diff sauberer und verhindert, dass Migrations-Bugs mit
Altlasten-Bugs verwechselt werden. Bewusst klein gehalten — keine grosse Architektur-Refactor,
die kommt (falls gewünscht) erst nach der Migration.

## Related To Intents

* 13 - Enhancement (Infrastruktur): Migration auf aktuelle Toolchain (dieser Intent geht voraus,
  damit der Migrations-Diff sauber bleibt)

## What

### Toter/auskommentierter Code
- `BuildMusicNetService#build`: auskommentierte Zeilen `#Track.delete_all`, `#Album.delete_all`,
  `#Artist.delete_all` (Orphan-Cleanup übernimmt das ohnehin am Ende der Methode)
- `UsersController#spotify`: ~30 Zeilen auskommentierter RSpotify-Beispielcode (Playlists, Follow,
  Top-Tracks) — Scaffolding-Reste
- `Playlist`-Model: auskommentierte alte `short_name`-Implementierung, durch die aktive Methode
  darunter ersetzt
- `Playlist`-Model: unbenutzte Konstante `COLORS_TO_CONTEXT`
- `config/initializers/omniauth.rb`: komplette Datei ist auskommentiertes Boilerplate, das dem
  echten Setup in `config/initializers/devise.rb` widerspricht — verwirrend gerade für den
  bevorstehenden Devise/omniauth-spotify-Major-Bump in Intent 13
- `app/views/playlist_tracks/_playlist_track.erb`: auskommentiertes `render partial:
  'components/spotify_player'`
- `spec/used_gems/rspotify_spec.rb`: Debug-`puts client_id` / `puts time` — leakt eine
  Credential-ID in Test-Logs

### Toter Code-Pfad
- `TracksController#play` + zugehörige Route (`member { get :play }`): laut eigenem Kommentar
  bekanntermassen defekt (`RestClient::Unauthorized`) und nirgends im Frontend verlinkt (kein
  `play_track_path`-Aufruf in irgendeiner View) — sicher komplett entfernbar

### Unbenutzter Code
- `spec/graphiti_helper.rb`: verwaiste Datei, requirt ein Gem (`graphiti_spec_helpers`), das nicht
  im Gemfile ist und von `rails_helper.rb` nicht geladen wird — Rest eines abgebrochenen
  Graphiti/JSON:API-Experiments

### Riskante Patterns
- `DownloadPlaylistService#download` / `DownloadTrackService#download` / `Track#track_path`:
  `Dir.chdir` jeweils ohne Block, ändert das Prozess-Arbeitsverzeichnis dauerhaft
- `Track#track_path`: `search = name; search.gsub!(...)` mutiert potenziell das `name`-Attribut
  des Tracks in-place (Aliasing-Bug)
- `Track#genre`: bare `rescue` (ohne Exception-Klasse) schluckt alles kommentarlos
- `config/application.rb` (`authenticate_to_spotify`): `puts`-Debugausgaben bei jedem Boot, in
  Dev **und** Prod
- `DownloadPlaylistService#initialize`: Parameter `user` wird nie verwendet
- `BuildMusicNetService::ServiceInfo#add`: behandelt `deleted`-Namen anders als `created`-Namen
  (verschachtelte statt flache Arrays) — inkonsistent, wird aktuell aber nirgends im UI ausgewertet
- `app/views/layouts/_navbar.html.erb`: `current_user.spotify_user.images.first["url"]` — Live-
  Spotify-API-Call auf jeder Seite, keine Fehlerbehandlung bei abgelaufenem Token
- `TracksController#stream`: kein Fallback/Template, wenn `track_path` nil ist
- `Gemfile`: totes `gem 'turbolinks'` neben `turbo-rails`/Stimulus (App nutzt tatsächlich Hotwire);
  `application.html.erb` nutzt noch das veraltete `data-turbolinks-track` statt `data-turbo-track`

### Config-Drift
- `config/application.rb`: `config.load_defaults 6.1`, obwohl die installierte Rails-Gem-Version
  bereits 7.1.1 ist. Da die Gem-Version schon da ist, ist das Nachziehen der Framework-Defaults
  auf 7.1 (schrittweise 6.1→6.1-frozen→7.0→7.1, mit Testsuite nach jedem Schritt) risikoarm und
  gehört in diese Aufräumrunde, nicht erst in die eigentliche Migration (Intent 13)
