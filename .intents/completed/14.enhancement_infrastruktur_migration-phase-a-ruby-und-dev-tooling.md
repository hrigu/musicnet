# 14 - Enhancement (Infrastruktur): Migration Phase A — Ruby und Dev-Tooling

## Why

Erste, risikoärmste Phase der in Intent 13 geplanten Migrations-Roadmap: Ruby-Interpreter und
reine Dev-/Test-Tooling-Gems sind unabhängig vom Rails-Major-Bump anhebbar.

## Depends On Intents

* 13 - Enhancement (Infrastruktur): Migration auf aktuelle Toolchain (übergeordnete Roadmap)

## What

- Ruby 3.2.2 → 3.4.10 (aktuellste verfügbare Version laut `rbenv install --list`, Ruby 3.5 noch
  nicht released)
- Dev-/Test-Tooling-Gems soweit angehoben, wie es die weiterhin gepinnte Rails-Version 7.1.1
  zulässt: rspec-rails 6.0.3→6.1.1, rubocop 1.57.1→1.88.1 (inkl. voller Abhängigkeitskette:
  rubocop-ast, parser, unicode-display_width, regexp_parser, ast), byebug→12.0.0,
  bootsnap→1.24.6, pry/pry-rails, listen→3.10.0, rack-mini-profiler→4.0.1, spring→4.7.0
- `ffi` 1.16.3→1.17.4 (siehe Challenges & Solutions — war Voraussetzung, damit `sassc`/`bootstrap`
  überhaupt unter Ruby 3.4 installierbar sind)

Bewusst **nicht** angehoben (durch Rails 7.1.1 blockiert, folgt in späteren Phasen): devise,
importmap-rails, omniauth-spotify, omniauth-rails_csrf_protection, rails selbst, sqlite3,
stimulus-rails, turbo-rails, sprockets-rails.

## Challenges & Solutions

- **`RBENV_VERSION` in dieser Agent-Session fest auf 3.2.2 gepinnt** (nicht aus einer Shell-Config
  der Nutzerin, sondern aus der Sandbox-Umgebung selbst) — überstimmt `.ruby-version`. Umgangen mit
  explizitem `RBENV_VERSION=3.4.10`-Prefix bei jedem Befehl. In einem normalen Terminal sollte
  `.ruby-version` automatisch greifen; falls nicht, prüfen, ob dort ebenfalls eine
  `RBENV_VERSION`-Env-Var gesetzt ist.
- **`ffi` 1.16.3 kompiliert nicht unter Ruby 3.4 / neuerem Xcode-Toolchain**: `bundle install`
  brach beim Kompilieren des vendored `libffi`-Assemblycodes ab (`invalid CFI advance_loc
  expression` in `aarch64/sysv.S`), verursacht durch `bootstrap` → `sassc-rails` → `sassc` → `ffi`.
  Gelöst durch `bundle update ffi sassc sassc-rails` — `ffi` 1.17.x bringt vorkompilierte
  `arm64-darwin`-Binaries mit, keine lokale Kompilierung mehr nötig.
- **`rubocop` 1.57.x kennt Ruby 3.4 nicht** (`RuboCop found unknown Ruby version: 3.4`, Absturz).
  `bundle update rubocop` allein reichte nicht (blieb bei 1.57.2, da `--conservative`-artiges
  Verhalten die Abhängigkeitskette nicht mitzog); erst `bundle update rubocop rubocop-ast parser
  unicode-display_width regexp_parser ast` explizit zusammen brachte 1.88.1.

## How

* [x] 1.0 - Aktuelle Ruby-Version ermitteln und installieren:
    - [x] 1.1 - `rbenv`/`ruby-build` aktualisieren (lokale Definitionsliste war veraltet)
    - [x] 1.2 - Ruby 3.4.10 installieren, `.ruby-version` und `Gemfile` (`ruby '3.4.10'`) anpassen
* [x] 2.0 - Bundler/Gems unter der neuen Ruby-Version neu aufbauen:
    - [x] 2.1 - Bundler-Version passend installieren, `bundle install` — `ffi`-Kompilierfehler
          gefunden und durch gezielten `ffi`/`sassc`-Bump gelöst
    - [x] 2.2 - Dev-/Test-Tooling-Gems angehoben (siehe Liste oben)
* [x] 3.0 - Verifikation:
    - [x] 3.1 - Testsuite unter Ruby 3.4.10 (47/47 grün)
    - [x] 3.2 - Frischer Boot-Check via `bin/rails runner` (inkl. Spotify-Authentifizierung beim
          Boot erfolgreich)
    - [x] 3.3 - Rubocop Lint/Security projektweit (keine Funde)
