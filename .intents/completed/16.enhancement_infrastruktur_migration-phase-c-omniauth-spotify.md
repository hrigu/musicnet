# 16 - Enhancement (Infrastruktur): Migration Phase C — omniauth-spotify 0.0.13 → 1.0.3

## Why

Dritte, riskanteste bisherige Phase der Migrations-Roadmap (Intent 13): `omniauth-spotify` ist ein
kleines, wenig gepflegtes Gem und steuert direkt den Spotify-Login — der einzige Auth-Weg der App.

## Depends On Intents

* 13 - Enhancement (Infrastruktur): Migration auf aktuelle Toolchain (übergeordnete Roadmap)

## What

`omniauth-spotify` 0.0.13 → 1.0.3. Quellcode beider Versionen direkt verglichen (0.0.13 lokal
entpackt via `gem fetch`/`gem unpack`, da kein öffentliches Changelog gepflegt wird):

- **`credentials`-Block neu hinzugefügt** (Access-/Refresh-Token, Ablaufzeit, Scope) — fehlte in
  0.0.13 komplett. `RSpotify::User` erwartet genau dieses Feld für spätere authentifizierte
  API-Calls (`recently_played` etc.). Vermutlich ein Bugfix, aber **nur durch echten Login
  end-to-end verifizierbar** — die gemockten Tests umgehen die Strategie komplett und bauen den
  `OmniAuth::AuthHash` selbst.
- `follower_count`: von `raw_info['followers']['total']` (crasht bei fehlendem `followers`-Key)
  auf sicheres `raw_info.dig("followers", "total")` — Bugfix.
- `birthdate`-Parsing: von ungeschütztem `Date.parse` auf `Date.iso8601` mit Rescue — robuster.
  URL-Struktur (`site` + Pfad) funktional identisch zu vorher.
- Restliche Änderungen (compact info-Hash, `query_string`-Override, Legacy-Key für Force-Approval)
  sind additive Robustheits-Verbesserungen ohne erkennbares Bruchpotenzial für unseren Code.

## Constraints

**Wichtig:** Diese Phase ist mit automatisierten Tests/Boot-Checks allein nicht vollständig
verifizierbar, weil der eigentliche OAuth2-Handshake (Redirect zu Spotify, Consent, Callback mit
echtem Code) nicht simulierbar ist. Vor produktivem Einsatz muss ein echter Spotify-Login im
Browser durchgeführt und geprüft werden, dass "Zuletzt gespielt" (recently_played) weiterhin
funktioniert — das ist genau die Funktion, die vom neuen `credentials`-Block abhängt.

## How

* [x] 1.0 - Quellcode-Diff zwischen 0.0.13 und 1.0.3 erstellen und auf Breaking Changes prüfen
* [x] 2.0 - omniauth-spotify auf 1.0.3 anheben (`bundle update`, konfliktfrei resolved)
* [x] 3.0 - Automatisierte Verifikation (Grenzen siehe Constraints):
    - [x] 3.1 - Testsuite (47/47 grün)
    - [x] 3.2 - Frischer Boot-Check + Routen-Check (`user_spotify_omniauth_callback` weiterhin
          vorhanden)
    - [x] 3.3 - Sign-in-Seite per simuliertem Request geprüft (200, Spotify-Link vorhanden)
    - [x] 3.4 - Rubocop Lint/Security (keine Funde)
* [x] 4.0 - Echter Spotify-Login im Browser durch den User bestätigt: "Zuletzt gespielt"
      funktioniert weiterhin, sogar nach Logout aus Spotify im Browser (Browser-Session und
      OAuth-Access-Token sind getrennt — der gespeicherte Token bleibt gültig, unabhängig vom
      Spotify-Browser-Login-Status). Bestätigt, dass der neue `credentials`-Block einen
      brauchbaren Access-Token liefert.
