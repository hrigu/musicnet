# 35 - Enhancement (Tracks-Audio-Features): Essentia statt Spotify für Tempo/Energy

## 1. Why

### 1.1 Objective

`Track#tempo`/`#energy` sollen aus lokal heruntergeladenen Audiodateien berechnet werden
(via der Bibliothek Essentia, CLI-Shellout), statt über Spotifys Audio-Features-Endpoint,
der für diese App dauerhaft gesperrt ist. Die alte, nutzlos gewordene Spotify-Lösung wird
vollständig entfernt.

### 1.2 Context

Spotify hat den `audio-features`-Endpoint seit 27.11.2024 für alle Apps ohne bereits
bestehenden Extended-Access gesperrt (403 Forbidden). Extended Quota Mode ist seit Mai 2025
nur noch für Businesses mit ≥250'000 MAU erreichbar — für diese App also dauerhaft
verschlossen, kein Workaround absehbar.

Nebenbefund bei der Analyse (2026-07-04): In der DB steht bei **allen** 2466 Tracks der
Literalwert `"null"` in `audio_features` — nicht `NULL`. Grund: `t.audio_features =
audio_features.to_json` (`build_music_net_service.rb`) schreibt bei fehlgeschlagenem Fetch
`nil.to_json` = `"null"` in eine bereits als `t.json` typisierte Spalte; ActiveRecord
kodiert das beim Schreiben ein zweites Mal JSON (daher der bestehende Kommentar zum
«doppelt JSON-kodiert» in `track.rb`). Es hat also nie echte Audio-Features-Daten aus
Spotify gegeben, die verloren gehen könnten — der Wechsel ist eine reine Neubefüllung.

### 1.3 Related to Intents

* 28 - Enhancement (Tracks-Track): Genre in DB cachen — liefert das Vorbild für den
  Read-Through-Cache-/Invalidierungs-Ansatz (Datei parsen, Ergebnis via `update_column`
  persistieren).
* 33 - Enhancement (Spotify-Playlists): Batch-API-Aufrufe beim Erstimport — hat den jetzt zu
  entfernenden Audio-Features-Prefetch eingeführt.
* 34 - Feature (Tracks-Index): Paginierung, Sortierung, Volltextsuche — `SORT_COLUMNS`
  sortiert per `json_extract` auf `audio_features`; die Spalten-Verschachtelung wird in
  diesem Intent vereinfacht (siehe 3.1).

## 2.0 What

```gherkin
Feature: Tracks Audio-Features - Essentia statt Spotify

  Als DJ
  möchte ich Tempo und Energy aus den heruntergeladenen Audiodateien berechnet bekommen
  damit diese Werte trotz gesperrtem Spotify-Endpoint wieder verfügbar sind.

  Scenario: Ein frisch heruntergeladener Track bekommt Tempo/Energy
    Given ein Track ohne audio_features, dessen Datei gerade heruntergeladen wurde
    When der Download abgeschlossen ist
    Then wird essentia_streaming_extractor_music für die Datei aufgerufen
    And tempo und energy werden aus dem Ergebnis in audio_features gespeichert

  Scenario: Essentia schlägt für einen Track fehl
    Given essentia_streaming_extractor_music liefert einen Fehler oder kein verwertbares Ergebnis
    When der Download abgeschlossen ist
    Then bleibt audio_features für diesen Track leer
    And der Fehler wird geloggt
    And der restliche Download-Vorgang läuft normal weiter

  Scenario: Bereits vorhandene Downloads nachträglich befüllen
    Given Tracks, deren Datei schon vor diesem Feature heruntergeladen wurde
    When der Rake-Task zur Nachbefüllung läuft
    Then bekommen alle Tracks mit vorhandener Datei und fehlenden audio_features Tempo/Energy

  Scenario: Kein Spotify-Request mehr für Audio-Features
    Given ein neuer Track wird beim Sync angelegt
    When BuildMusicNetService läuft
    Then wird kein Request mehr an RSpotify::AudioFeatures gestellt
```

## 3.0 How

### 3.1 Implementation Context

* **Essentia als CLI, nicht als FFI-Bindung:** Essentia hat keine stabile `extern "C"`-
  Shared-Lib-API zum Binden (nur Python-/JS(WASM)-Bindings offiziell). Genutzt wird
  stattdessen das mitgelieferte CLI-Tool `essentia_streaming_extractor_music
  <input> <output>` (Installation z. B. via `brew install essentia`, wie `spotdl` nicht
  gebundelt, muss auf PATH liegen). Es schreibt standardmässig YAML — das wird direkt mit
  Rubys `YAML.safe_load_file` geparst, kein zusätzliches Gem nötig. Verwendet werden nur
  zwei Felder aus dem Output: `rhythm.bpm` (Tempo) und `lowlevel.average_loudness`
  (0..1-normalisiertes Lautheits-Mass, als Energy-Analogon — konzeptionell nicht identisch
  mit Spotifys ML-basiertem `energy`, aber die nächstliegende lokal berechenbare Grösse).
  Andere Spotify-Felder (danceability, valence, key, ...) sind bewusst **nicht** Teil
  dieses Intents.
* **Hash statt `.to_json`-String schreiben:** Behebt nebenbei den unter 1.2 gefundenen
  Doppel-Encoding-Bug. `audio_features` ist eine `t.json`-Spalte — ActiveRecord
  (de-)serialisiert sie bereits automatisch. Ein Hash `{ "tempo" => ..., "energy" => ... }`
  wird direkt zugewiesen, `Track#af` liest `audio_features` künftig direkt als Hash (kein
  manuelles `JSON.parse` mehr), `SORT_COLUMNS["energy"/"tempo"]` braucht nur noch ein
  einfaches `json_extract` statt der bisherigen doppelten Verschachtelung.
* **Hook-Punkt:** `DownloadPlaylistService#download` ist der einzige Konvergenzpunkt beider
  Download-Wege (`DownloadTrackService` delegiert an ihn playlist-weise). Nach
  erfolgreichem `system(...)`-Aufruf wird ein neuer `AudioFeaturesExtractionService` für
  `@playlist.tracks` aufgerufen, der (via `Track.preload_track_paths`) nur Tracks mit
  vorhandener Datei und noch leerem `af` verarbeitet.
* **Bestandsaufnahme:** Für die 2466 bereits heruntergeladenen Tracks ein Rake-Task
  (analog `backfill_album_and_artist_details`), der dieselbe Extraction-Logik einmalig
  über alle Tracks mit vorhandener Datei laufen lässt.
* **Datenbereinigung:** Migration, die `audio_features` für alle bestehenden Zeilen auf
  `NULL` zurücksetzt (reine Datenmigration, keine Schemaänderung nötig) — sonst würde
  `Track#af` beim ersten Aufruf nach dem Umbau auf dem alten String-Literal `"null"`
  crashen (kein Hash, `OpenStruct.new` erwartet eines).
* **Performance/Scope-Grenze:** Essentias volle Musik-Extraktion dauert pro Track spürbar
  länger als der reine Download (mehrere Sekunden CPU-Zeit für Spektral-/Rhythmus-Analyse).
  Sie läuft synchron innerhalb des bestehenden Download-Locks — es gibt im Projekt keine
  Background-Job-Infrastruktur (kein Sidekiq/ActiveJob-Adapter), ein Async-Umbau ist
  bewusst nicht Teil dieses Intents.
* **Tests:** `system`-Aufrufe an `essentia_streaming_extractor_music` sind externe
  Boundaries und werden wie bei `DownloadPlaylistService` üblich auf der Service-Instanz
  gestubbt (`allow(service).to receive(:system)`), keine echten Prozessaufrufe in Specs.
