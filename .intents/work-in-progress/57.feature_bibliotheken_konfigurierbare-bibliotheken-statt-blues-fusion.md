# 57 - Feature (Bibliotheken): Konfigurierbare Bibliotheken statt hardcoded Blues/Fusion

## 1. Why

### 1.1 Objective

Sowohl der Spotify-Import-Filter (`SpotifyPlaylistsGateway`, hardcoded `/fusion|blues/i`) als auch
der Anzeige-Filter (Intent 54, fester Enum "all"/"blues"/"fusion") kennen bisher nur genau zwei,
im Code fest verdrahtete Kategorien. Der DJ hat aber mehr Playlists als nur Fusion/Blues und will
frei benennbare "Bibliotheken" anlegen können, denen Playlists automatisch zugeordnet werden — eine
Playlist kann dabei zu mehreren Bibliotheken gleichzeitig gehören (z.B. eine Playlist, deren Name
sowohl "Blues" als auch "Fusion" enthält).

### 1.2 Context

Aktuell fallen zwei technisch getrennte, aber inhaltlich verwandte Mechanismen zusammen:
* **Import-Filter** (`SpotifyPlaylistsGateway#owned_fusion_or_blues_playlist?`): entscheidet beim
  Sync, welche Spotify-Playlists überhaupt lokal gespiegelt werden.
* **Anzeige-Filter** (Intent 54, `User#active_playlist_category`/`active_category_substring`,
  Scopes `Playlist.in_active_category`/`Track.in_active_category`/`Artist.in_active_category`):
  schränkt rein die Anzeige ein, ändert nichts am Sync.

Beide arbeiten heute per Namens-Teilstring-Vergleich gegen dieselben zwei hardcoded Wörter. Dieser
Intent ersetzt das durch ein neues Model `Library` (Name + Stichwort), das beide Mechanismen
speist: das Stichwort entscheidet beim Sync automatisch, ob und welchen Bibliotheken eine Playlist
zugeordnet wird; der Anzeige-Filter wählt eine aktive Bibliothek (oder "Alle") aus den vorhandenen
Libraries statt aus einem festen Enum.

**Bewusst nicht Teil dieses Intents:** manuelles Überschreiben der automatischen
Stichwort-Zuordnung (z.B. falls eine Playlist trotz fehlendem Stichwort einer Bibliothek angehören
soll) — als Idee in `.intents/Ideen.md` festgehalten, falls sich die reine Stichwort-Logik als zu
unflexibel erweist.

### 1.3 Depends On Intents

* 31/33 - Feature (Spotify-Playlists): Sync-Grundlagen (`BuildMusicNetService`,
  `SpotifyPlaylistsGateway`)
* 43 - Feature (Tracks-Suche): DSL-Suche mit Autocomplete (`TrackQuerySuggestions`)
* 54 - Feature (Anzeige): Aktive Kategorie als Anzeige-Filter — wird durch diesen Intent ersetzt
* 55 - Bug (Tracks-Suche-Autocomplete): nutzt den Anzeige-Filter für Vorschläge

## 2.0 What

```gherkin
Feature: Bibliotheken - Konfigurierbare Bibliotheken statt hardcoded Blues/Fusion

  Als DJ
  möchte ich frei benennbare Bibliotheken mit eigenem Stichwort anlegen können
  damit Playlists automatisch danach einsortiert werden, statt nur nach Blues/Fusion

  Scenario: Bestehende Daten bleiben nach der Migration unverändert
    Given ich hatte vor der Migration die Kategorie "Nur Blues" aktiv
    When die Migration läuft
    Then existieren die Bibliotheken "Blues" (Stichwort "blues") und "Fusion" (Stichwort "fusion")
    And alle bereits importierten Playlists sind nach ihrem Namen den passenden Bibliotheken zugeteilt
    And meine aktive Bibliothek ist weiterhin "Blues"

  Scenario: Eine neue Bibliothek anlegen
    Given ich bin auf der Bibliotheken-Seite
    When ich eine Bibliothek "Deep House" mit Stichwort "house" anlege
    Then erscheint "Deep House" in der Liste der Bibliotheken

  Scenario: Sync importiert nach konfigurierbarem Stichwort
    Given es existiert eine Bibliothek "Deep House" mit Stichwort "house"
    And ich besitze eine Spotify-Playlist "Deep House Vibes", die weder "blues" noch "fusion" heisst
    When ich "Alle Playlists holen" ausführe
    Then wird "Deep House Vibes" importiert und der Bibliothek "Deep House" zugeteilt

  Scenario: Eine Playlist gehört zu mehreren Bibliotheken
    Given es existieren die Bibliotheken "Blues" (Stichwort "blues") und "Fusion" (Stichwort "fusion")
    And ich besitze eine Spotify-Playlist "Blues Fusion Night"
    When ich "Alle Playlists holen" ausführe
    Then ist "Blues Fusion Night" sowohl der Bibliothek "Blues" als auch "Fusion" zugeteilt

  Scenario: Umbenennung auf Spotify aktualisiert die Zuordnung
    Given eine lokale Playlist ist aktuell nur der Bibliothek "Blues" zugeteilt
    And sie wird auf Spotify in einen Namen umbenannt, der zusätzlich "fusion" enthält
    When die Playlist das nächste Mal synchronisiert wird
    Then ist sie danach sowohl "Blues" als auch "Fusion" zugeteilt

  Scenario: Anzeige-Filter zeigt nur die aktive Bibliothek
    Given die aktive Bibliothek ist "Deep House"
    When ich `/tracks`, `/playlists` oder `/artists` aufrufe
    Then sehe ich nur Einträge aus Playlists der Bibliothek "Deep House"

  Scenario: Eine Bibliothek löschen, die gerade aktiv ist
    Given meine aktive Bibliothek ist "Deep House"
    When ich die Bibliothek "Deep House" lösche
    Then ist meine aktive Bibliothek danach wieder "Alle"
```

## 3.0 How

### 3.1 Implementation Context

**Neue Models:**
* `Library` (`name:string`, `keyword:string`, beide `presence`-validiert, `name` `uniqueness`-
  validiert) mit `Library.matching(playlist_name)` als einzige Stelle, die den
  Stichwort-Teilstring-Vergleich (case-insensitive) durchführt — sowohl der Import-Filter als auch
  die automatische Zuordnung rufen ausschliesslich diese Methode auf, keine zweite Kopie der Logik.
* `LibraryPlaylist` (reines Join-Model, `belongs_to :library`, `belongs_to :playlist`, Unique-Index
  auf `[library_id, playlist_id]`).
* `Playlist has_many :libraries, through: :library_playlists` (n:m, ersetzt keine bestehende
  Assoziation).

**Migration/Datenmigration (eine Migration, mehrere Schritte):**
1. Tabellen `libraries`/`library_playlists` anlegen.
2. Zwei Library-Zeilen seeden: "Blues"/"blues", "Fusion"/"fusion".
3. Für jede bestehende lokale `Playlist` `LibraryPlaylist`-Zeilen anhand `Library.matching(name)`
   nachtragen (bestehende Daten bleiben äquivalent zum heutigen Verhalten).
4. `users.active_library_id` (FK, nullable) hinzufügen; bestehende
   `active_playlist_category`-Werte übersetzen (`"blues"` → Blues-Library-Id, `"fusion"` →
   Fusion-Library-Id, `"all"`/unerwartet → `nil`), dann Spalte `active_playlist_category` entfernen.

**Import-Filter (`SpotifyPlaylistsGateway`):** `owned_fusion_or_blues_playlist?` wird zu
`owned_library_playlist?(playlist) = playlist.owner.id == @spotify_user.id &&
Library.matching(playlist.name).any?`.

**Automatische Zuordnung (`BuildMusicNetService`):** neue private Methode `assign_libraries
(playlist)` setzt `playlist.library_ids = Library.matching(playlist.name).map(&:id)` (Rails
gleicht die n:m-Zeilen bei `collection_ids=` automatisch ab — fügt neue hinzu, entfernt fehlende).
Aufgerufen am Ende von `build_playlist` (nach dem Erstellen) und am Ende von
`sync_playlist_with_spotify` (nach `playlist.update!`, da sich der Name geändert haben kann).

**Bibliotheken-Verwaltung:** neuer `LibrariesController` (`resources :libraries`, Name+Stichwort),
Views analog zum bestehenden CRUD-Stil (z.B. `settings` oder `playlists`). Löschen einer Library
nullt betroffene `User#active_library_id` mit (Model-Association `dependent: :nullify` auf der
Kehrseite, oder Controller-seitig vor `destroy`).

**Anzeige-Filter (ersetzt Intent 54):** `Playlist.in_active_category`/`Track.in_active_category`/
`Artist.in_active_category` werden zu `in_active_library(library_id)` (gleiches
Subquery-`where(id: ...)`-Muster wie bisher, blank `library_id` → ungefiltert). Settings-View zeigt
eine Radiobutton-Liste "Alle" + `Library.order(:name)` statt der drei festen Optionen.
`TracksController#index`/`PlaylistsController#index`/`ArtistsController#index` sowie
`TrackQuerySuggestions` (Intent 55, Playlist-/Artist-/Genre-Vorschläge) werden auf
`current_user.active_library_id` umgestellt.

**Playlist-Anzeige:** Playlist-Index/-Show zeigt die zugeordneten Library-Namen an (rein
informativ, da automatisch berechnet, keine manuelle Bearbeitung hier).
