# 27 - Enhancement (Tracks-Track): Index-Seite ohne Query- und Glob-Lawine rendern

## 1. Why

### 1.1 Objective

Die Track-Übersicht (`tracks#index`) soll in wenigen Sekunden rendern statt in ~23 Sekunden,
indem pro Request genau ein Verzeichnis-Scan und eine handvoll gebündelte SQL-Queries laufen.

### 1.2 Context

Log eines Aufrufs (2026-07-03 07:12): `Completed 200 OK in 23096ms (Views: 23079.8ms)`.
Bei 2466 Tracks und 2270 Dateien in `downloads/tracks` gibt es zwei Ursachen:

1. **Glob-Lawine**: `Track#genre` ruft `track_path` zweimal auf (Guard + `WahWah.open`).
   Jeder Aufruf macht ein `Dir.glob` über das ganze Verzeichnis (~4ms). Das sind
   2 × 2466 Globs ≈ **19s** (Benchmark: 200 × track_path = 782ms). Das eigentliche
   Metadaten-Lesen mit WahWah kostet nur ~1.7s gesamt.
2. ~~**N+1-Queries**~~ **Korrektur während der Implementation**: Der vermutete N+1 auf
   `playlist_tracks`/`playlists` existiert nicht — Rails preloadet bei `includes(:playlists)`
   (has_many :through) die Zwischen-Association samt Quelle gleich mit. Die 1802 Queries im
   Log stammten von einem parallel laufenden Sync (`BuildMusicNetService`), dessen Logzeilen
   sich mit dem Request überlagerten; saubere Aufrufe zeigen 7 Queries. Task 1 sichert das
   gebündelte Laden nur noch als Regressions-Spec ab.

`tracks#download` hat dieselbe Glob-Lawine: eine Schleife über `Track.all` mit `track_path`
pro Track (~10s nur fürs Sammeln der fehlenden Dateien).

## 2.0 What

```gherkin
Feature: Tracks Track - Index-Seite performant rendern

  Als DJ
  möchte ich dass die Track-Übersicht ihre Daten gebündelt lädt
  damit die Seite in Sekunden statt in einer halben Minute rendert.

  Scenario: Playlist-Badges ohne Einzel-Queries
    Given mehrere Tracks mit Playlist-Zuordnungen
    When ich die Track-Übersicht aufrufe
    Then werden die Badges korrekt angezeigt
    And es wird keine Query pro Track für playlist_tracks oder playlists ausgeführt

  Scenario: Ein Verzeichnis-Scan pro Request
    Given mehrere Tracks mit teilweise vorhandenen Audio-Dateien
    When ich die Track-Übersicht aufrufe
    Then wird das Download-Verzeichnis genau einmal gelesen
    And track_path liefert dieselben Pfade wie bisher (inkl. nil für fehlende Dateien)
```

## 3.0 How

### 3.1 Implementation Context

Kleine, fokussierte Eingriffe nach dem Muster von Intent 22/24 — TDD über Request- und
Model-Specs, Query-Zählung per `ActiveSupport::Notifications` auf `sql.active_record`:

1. **Badge-Queries absichern**: Das bestehende `includes(:artists, :playlists, :album)` lädt
   bereits alles gebündelt (siehe Korrektur oben) — ein Regressions-Spec zählt die Queries
   pro Tabelle, damit ein künftiger Umbau des `includes` den Bündel-Load nicht still verliert.
2. **Verzeichnis-Scan bündeln**: Neue Klassenmethode `Track.preload_track_paths(tracks)` liest
   das Verzeichnis einmal (`Dir.children`) und setzt auf jedem Track den memoisierten Pfad.
   `Track#track_path` memoisiert sein Resultat (`defined?`-Guard, da nil ein gültiger Wert ist)
   — das behebt auch den Doppel-Glob in `genre`. Das In-Memory-Matching repliziert die bisherige
   Glob-Semantik `*-?<name>.m4a` exakt: Dateiname endet auf `<name>.m4a`, davor ein beliebiges
   Zeichen, davor `-` (dieselben Zeichenersetzungen wie bisher, aber ohne Glob-Escapes für `[`/`]`).
   `tracks#index` und `tracks#download` rufen das Preload auf; Einzel-Track-Stellen
   (`show`, `stream`, Rake-Task) bleiben beim Glob-Fallback.

Erwarteter Effekt: ~23s → ~2–3s (verbleibend: WahWah-Metadaten ~1.7s + Rendering der 2466 Zeilen).
Weitergehende Ideen (Genre in DB cachen, Pagination) bewusst ausserhalb dieses Intents.
