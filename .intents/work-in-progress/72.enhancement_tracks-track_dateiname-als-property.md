# 72 - Enhancement (Tracks-Track): Dateiname als DB-Spalte cachen statt Namens-Matching

## 1. Why

### 1.1 Objective

`Track#file_name` soll den tatsächlich von `spotdl` vergebenen Dateinamen einmalig nach dem Download in einer eigenen Spalte speichern, statt ihn bei jedem Zugriff über `TrackFileLocator` per Namens-Suffix-Matching gegen `downloads/tracks` neu zu erraten.

### 1.2 Context

`TrackFileLocator.resolve_track_path` matcht aktuell ausschliesslich über den Track-**Namen** als Datei-Suffix (`-<name>.m4a`), ohne Artist oder Spotify-ID einzubeziehen. Haben zwei Tracks exakt denselben Namen (z.B. "Blues" von zwei verschiedenen Artists), erzeugen beide denselben Such-Suffix — `file_entries.find` liefert dann für beide Track-Records dieselbe (alphabetisch erste) Datei zurück. Das führt zu einer stillen Fehlzuordnung: falscher Song beim Abspielen, falsches Genre/falsche Audio-Features (beide aus der Datei gelesen). Eine feste DB-Spalte, einmal beim Download aus dem tatsächlich aufgelösten Pfad gesetzt, macht das erneute Raten überflüssig und behebt die Kollision für alle ab jetzt neu heruntergeladenen Tracks. Für bereits heruntergeladene Tracks füllt ein Rake-Task die Spalte rückwirkend über den bestehenden (fehleranfälligen) Matching-Weg — bereits bestehende Namenskollisionen in historischen Daten werden damit nicht rückwirkend aufgelöst (siehe 3.1, bewusste Einschränkung).

### 1.3 Depends On Intents

* 27 - Enhancement (Tracks-Track): führte den gebündelten Verzeichnis-Scan (`preload_track_paths`) ein, an dem sich der Fallback-Pfad weiterhin orientiert.
* 28 - Enhancement (Tracks-Track): etablierte das DB-Cache-Muster (`update_column`, manuelle Invalidierung), auf dem `file_name` aufbaut.
* 38 - Feature (Downloads-Playlist-Download): `DownloadResultParser`, der `track_path` bereits frisch nach jedem Download neu auflöst — hier wird `file_name` zusätzlich persistiert.

## 2.0 What

```gherkin
Feature: Tracks-Track - Dateiname als Property

  Als DJ
  möchte ich, dass jeder Track seinen heruntergeladenen Dateinamen fest speichert
  damit zwei gleichnamige Tracks nie mehr versehentlich dieselbe Datei zugewiesen bekommen.

  Scenario: Dateiname wird nach einem Download persistiert
    Given ein Track wurde gerade erfolgreich heruntergeladen
    Then wird sein tatsächlicher Dateiname in Track#file_name gespeichert

  Scenario: Track-Pfad wird direkt aus der gespeicherten Property aufgelöst
    Given ein Track hat bereits eine gespeicherte file_name
    When sein track_path abgefragt wird
    Then wird die Datei direkt anhand von file_name gefunden, ohne Namens-Matching

  Scenario: Zwei gleichnamige Tracks bleiben unterscheidbar
    Given zwei Tracks mit exakt demselben Namen wurden je einzeln heruntergeladen
    Then hat jeder seine eigene, korrekte file_name und damit seinen eigenen track_path

  Scenario: Fallback für noch nicht migrierte Tracks
    Given ein bereits heruntergeladener Track hat noch keine file_name gespeichert
    When sein track_path abgefragt wird
    Then greift weiterhin das bisherige Namens-Matching als Fallback

  Scenario: Rückwirkendes Befüllen bestehender Tracks
    Given viele Tracks wurden vor Einführung dieses Features heruntergeladen
    When der Backfill-Rake-Task läuft
    Then bekommen alle per bisherigem Matching auflösbaren Tracks ihre file_name gesetzt
```

## 3.0 How

### 3.1 Implementation Context

* **Migration**: `add_column :tracks, :file_name, :string` (nullable, blank/nil = "noch nicht bekannt", analog zum bestehenden `genre`/`audio_features`-Nil-Muster).
* **`TrackFileLocator.resolve_track_path`**: prüft zuerst `track.file_name` — wenn gesetzt und `downloads_dir.join(file_name)` existiert (`File.exist?`, kein Verzeichnis-Scan nötig), wird der Pfad direkt zurückgegeben. Sonst (blank, oder Datei wurde gelöscht/umbenannt) Fallback auf das bestehende Suffix-Matching gegen `file_entries` wie bisher.
* **`TrackFileLocator.preload_track_paths`**: unverändert im Aufbau (ein Verzeichnis-Scan für den ganzen Batch), aber `resolve_track_path` pro Track nutzt jetzt zuerst den schnellen `file_name`-Pfad, bevor es überhaupt in die gescannte Liste schaut.
* **Neue Methode `Track#persist_file_name!`** (oder direkt inline in `DownloadResultParser`): setzt `file_name` per `update_column` aus `File.basename(track_path)`, nur wenn abweichend vom aktuellen Wert (kein unnötiger Write).
* **`DownloadResultParser#downloaded_entries`**: nach dem bestehenden `Track.preload_track_paths(@tracks)` zusätzlich für jeden Track mit vorhandenem `track_path` die `file_name` persistieren — einziger Ort, an dem frisch heruntergeladene Tracks laufen, sowohl für den Playlist-Download als auch für `DownloadMissingTracksJob` (der intern denselben Service nutzt).
* **Neuer Rake-Task `backfill_track_file_names`** (analog zu `extract_missing_audio_features`): iteriert alle Tracks mit `file_name: nil`, ruft das bestehende Matching auf (`TrackFileLocator.resolve_track_path` ohne gesetzte `file_name`, faktisch der alte Pfad) und setzt `file_name`, falls eine Datei gefunden wurde. Bekannte Einschränkung (dokumentiert im Task-Kommentar): bei historischen Namenskollisionen bekommen ggf. mehrere Tracks dieselbe (falsche) Datei zugewiesen, exakt wie im heutigen Verhalten — das lässt sich nur durch manuelle Prüfung der Betroffenen auflösen, nicht automatisch.
* **Tests**: `TrackFileLocator`-Spec erweitert um "nutzt `file_name` direkt, wenn vorhanden und Datei existiert", "fällt zurück aufs Namens-Matching, wenn `file_name` fehlt", "fällt zurück, wenn die per `file_name` referenzierte Datei nicht mehr existiert". `DownloadResultParserSpec`: nach `parse` haben erfolgreich heruntergeladene Tracks ihre `file_name` gesetzt. Model- oder Rake-Task-Spec für den Backfill (setzt `file_name` für auflösbare Tracks, lässt nicht auflösbare `nil`).
