# 13 - Enhancement (Infrastruktur): Migration auf aktuelle Toolchain

## Why

Ruby 3.2.2, Rails-Gem 7.1.1 (Framework-Defaults sogar nur auf Stand 6.1 via
`config.load_defaults 6.1`), Devise 4.x, omniauth-spotify 0.0.13, sqlite3-Gem 1.6.7 — alles
mehrere Major-Versionen hinter dem aktuellen Stand. Rails 7.1 ist inzwischen aus dem
Security-Support gefallen. Je länger gewartet wird, desto grösser der Sprung und das Risiko.

## Depends On Intents

* 12 - Enhancement (Infrastruktur): Altlasten aufräumen vor der Migration (sollte zuerst
  abgeschlossen sein, damit der Migrations-Diff sauber bleibt)

## What

Ziel-Stack: aktuelles Ruby 3.4.x/3.5.x (genaue Patch-Version zum Zeitpunkt der Umsetzung prüfen,
lokale `rbenv`-Definitionsliste war beim Verfassen dieses Intents veraltet), Rails 8.1.x,
`load_defaults 8.1`, Devise 5.x, omniauth-spotify 1.0.x, sqlite3-Gem 2.x, Rack 3.

Stand aus `bundle outdated` (Zeitpunkt der Erstellung dieses Intents):

| Gem | Aktuell | Ziel | Risiko |
|---|---|---|---|
| ruby | 3.2.2 | 3.4.x/3.5.x | niedrig |
| rails | 7.1.1 | 8.1.3 | hoch (Major, Rack 2→3) |

## Constraints

Kein CI vorhanden — jede Phase braucht manuelle Verifikation (inkl. echtem Spotify-OAuth-Login,
`spotdl`-Download, Mixxx-Export) zusätzlich zur Testsuite. Einzelbetrieb (kein
produktionskritischer Multi-User-Service), daher sequenziell/vorsichtig statt big-bang.

## Challenges & Solutions

- **sqlite3-Gem 2.x ist NICHT unabhängig von Rails 8 anhebbar** (ursprünglich als eigene
  "Phase B" vor dem Rails-Bump geplant). Rails 7.1.1s SQLite3-Adapter lädt intern
  `gem "sqlite3", "~> 1.4"` fest — ein `bundle update sqlite3` auf 2.9.5 resolved zwar
  konfliktfrei, aber die App crasht beim Boot mit `LoadError: can't activate sqlite3 (~> 1.4),
  already activated sqlite3-2.9.5`. Offizielle sqlite3-2.x-Unterstützung kam erst mit Rails 7.2/8.
  **Korrektur der Roadmap:** sqlite3 1.x→2.x wandert von einer eigenen frühen Phase in die
  Rails-8-Phase (jetzt Phase D unten), muss zusammen mit dem Rails-Bump passieren. Bis dahin
  `gem 'sqlite3', '~> 1.4'` explizit im Gemfile gepinnt (verhindert versehentliches
  `bundle update` auf 2.x durch einen künftigen breiten Update-Lauf).

## How

Jede Phase wird bei Umsetzung als eigener, kleinerer Intent ausgeplant. Hier nur die
Reihenfolge/Roadmap (korrigiert nach dem sqlite3-Fund):

* [x] 1.0 - Sicherheitsnetz prüfen: bestehende Testsuite (47 Tests, Intents 7-11) muss vor jeder
      Phase grün sein und bleiben — Migrationsfortschritt wird daran gemessen (47/47 grün bestätigt
      vor Start von Phase A)
* [x] 2.0 - Phase A (niedriges Risiko, unabhängig): siehe Intent 14 — Ruby 3.2.2→3.4.10, Dev-/
      Test-Tooling soweit angehoben, wie Rails 7.1.1 es zulässt. Unerwartete Komplikation gelöst:
      `ffi`/`sassc` kompilierte nicht unter Ruby 3.4, gefixt durch gezielten `ffi`-Bump auf 1.17.x.
      sqlite3-Bump auf 2.x versucht, als von Rails 7.1 blockiert erkannt, sqlite3 explizit auf
      `~> 1.4` gepinnt (siehe Challenges & Solutions) — sqlite3 2.x wandert zu Phase D
* [x] 3.0 - Phase B: siehe Intent 15 — Devise 4.9.3 → 5.0.4, keine Breaking Changes betroffen
      (keine eigenen Devise-Views, keine entfernten APIs im Einsatz), inkl. zweier
      Security-Fixes (CVE-2026-32700, CVE-2026-40295)
* [~] 4.0 - Phase C: siehe Intent 16 — omniauth-spotify 0.0.13 → 1.0.3, Quellcode-Diff erstellt
      (neuer `credentials`-Block, zwei Bugfixes). Automatisiert verifiziert, aber **echter
      Spotify-Login im Browser durch dich steht noch aus** (prüft insbesondere, ob "Zuletzt
      gespielt" mit dem neuen `credentials`-Block weiterhin funktioniert)
* [ ] 5.0 - Phase D: Rails 7.1 → 8.x **zusammen mit** sqlite3 1.x → 2.x (grösster Schritt: Rack
      2→3, `load_defaults 7.1→8.1`, importmap/turbo/stimulus im Gleichschritt aktualisieren,
      `bin/rails server`-Startverhalten prüfen)
* [ ] 6.0 - Phase E: Aufräumen — verbleibende Deprecation-Warnungen beseitigen,
      `.ruby-version`/Gemfile/README aktualisieren
