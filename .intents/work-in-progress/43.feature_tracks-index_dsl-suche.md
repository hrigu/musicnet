# 43 - Feature (Tracks-Index): DSL-basierte Textsuche (BPM, Genre, Künstler, Playlist-Schnittmenge, ...)

## 1. Why

### 1.1 Objective

Das bestehende Suchfeld auf `/tracks` (`q`, reine Volltextsuche über Name/Künstler/Album/Genre/
Playlist-Name) soll eine kleine Abfragesprache verstehen, mit der gezielt nach einzelnen
Track-Attributen gefiltert werden kann — z.B. `genre:jazz,fusion bpm:80..100 playlist:"Fusion
Abende" playlist:"Blues Session"`. Keine Klick-/Checkbox-UI, ein einziges Textfeld.

### 1.2 Context

`TracksController#index` (`app/controllers/tracks_controller.rb:13`) nutzt aktuell `Track.search`
(reiner LIKE-Substring über mehrere Spalten). Sortierung (`Track.sorted`) und Verfügbarkeits-Filter
(`available`) bleiben unverändert bestehen — nur die Freitextsuche wird durch die neue DSL ersetzt.

Recherche zu bestehenden Lösungen (siehe Chat-Verlauf) ergab:
* GitHub-/Gmail-Suchsyntax als Vorbild für `feld:wert`, Komma = ODER, Ranges via `..`/`>`/`>=`/`<`/
  `<=`, Anführungszeichen für Werte mit Leerzeichen.
* Das Ruby-Gem `search_cop` bietet eine sehr ähnliche String-DSL, ist aber auf MySQL/Postgres-
  Volltextindizes ausgelegt (SQLite nirgends als unterstützter Adapter dokumentiert) — Risiko für
  dieses Projekt (SQLite). `scoped_search` (ähnliche DSL) ist seit 2017 unmaintained.
* **Entscheidung:** kein externes Gem, ein kleiner eigener Parser — passt zum bestehenden Muster
  im Projekt (`Track::SORT_COLUMNS`: whitelist-basierte, handgeschriebene SQL-Fragmente inkl.
  `json_extract` für `audio_features`).

### 1.3 Depends On Intents

* 34 - Feature (Tracks-Index): Paginierung, Sortierung, Volltextsuche — Turbo-Frame-/
  Query-Param-Infrastruktur und `Track.search` (bleibt als Freitext-Baustein bestehen) werden
  wiederverwendet.
* 35 - Enhancement (Tracks-Audio-Features): Essentia statt Spotify — liefert `tempo`/`energy`.
* 28 - Enhancement (Tracks-Track): Genre in DB cachen.

### 1.4 Related to Intents

## 2.0 What

```gherkin
Feature: Tracks-Index - DSL-basierte Textsuche

  Als DJ
  möchte ich in einem einzigen Textfeld nach Attributen wie Genre, BPM, Künstler oder
  Playlist-Zugehörigkeit filtern können
  damit ich ohne Klick-UI gezielt z.B. Jazz-verwandte Stücke zwischen 80 und 100 BPM finde, die in
  zwei bestimmten Playlists gleichzeitig vorkommen.

  Scenario: Freitext wie bisher
    Given Tracks mit unterschiedlichen Namen, Künstlern, Alben und Genres
    When ich einen Suchbegriff ohne "feld:"-Präfix eingebe
    Then verhält sich die Suche wie die bestehende Volltextsuche (Intent 34)

  Scenario: Nach einem Feld filtern
    Given Tracks mit unterschiedlichem Genre
    When ich "genre:jazz" eingebe
    Then sehe ich nur Tracks mit diesem Genre

  Scenario: ODER innerhalb eines Feldes über Komma
    When ich "genre:jazz,fusion,blues" eingebe
    Then sehe ich Tracks, deren Genre einem der drei Werte entspricht

  Scenario: UND über mehrere Vorkommen desselben Feldes (Playlist-Schnittmenge)
    Given ein Track ist in Playlist A und B, ein anderer nur in Playlist A
    When ich "playlist:A playlist:B" eingebe
    Then sehe ich nur den Track, der in beiden Playlists vorkommt

  Scenario: Zahlen-/Datums-Range
    Given Tracks mit unterschiedlichem Tempo
    When ich "bpm:80..100" eingebe
    Then sehe ich nur Tracks mit Tempo zwischen 80 und 100 (inklusive Grenzen)

  Scenario: Vergleichsoperatoren
    When ich "popularity:>50" eingebe
    Then sehe ich nur Tracks mit Popularity grösser 50

  Scenario: Negation
    When ich "-genre:blues" eingebe
    Then sehe ich keine Tracks mit Genre Blues

  Scenario: Werte mit Leerzeichen in Anführungszeichen
    When ich 'playlist:"Fusion Abende"' eingebe
    Then wird "Fusion Abende" als ein zusammenhängender Wert behandelt

  Scenario: Mehrere Kriterien kombiniert
    When ich "genre:jazz,fusion bpm:80..100 playlist:Fusion" eingebe
    Then werden alle Kriterien mit UND verknüpft angewendet (ausser dem Komma-ODER innerhalb eines
      Feldes)

  Scenario: Unbekanntes Feld oder ungültiger Wert
    Given ich gebe "composer:Bach" oder "bpm:abc" ein
    When die Seite lädt
    Then gibt es keinen Fehler; ein unbekanntes Feld wird als Freitext behandelt, ein ungültiger
      Wert für ein bekanntes Feld wird ignoriert

  Scenario: Kombination mit Sortierung und Verfügbarkeits-Filter bleibt wie bisher
    Given ich nutze zusätzlich sort/direction/available
    When ich die DSL-Suche benutze
    Then werden weiterhin alle bestehenden Kriterien korrekt kombiniert (Intent 34 unverändert)

  Scenario: Nur Tabelle aktualisiert sich
    When ich die Suche ändere
    Then wird nur der Turbo Frame der Tabelle aktualisiert (wie bisher)
```

## 3.0 How

### 3.1 Implementation Context

* **Parser (`app/services/track_query_parser.rb`):** zerlegt den rohen String in Tokens
  (Leerzeichen-getrennt, Anführungszeichen schützen Leerzeichen im Wert). Ein Token ist entweder
  Freitext oder `[-]feld:wert`. Reine Parsing-Logik, keine DB-Zugriffe — gut isoliert testbar ohne
  Fixtures.
* **Wert-Klassifizierung pro Feld-Token:**
  * Komma im Wert → Liste (ODER, z.B. für `IN`)
  * `min..max` → Range
  * Präfix `>`, `>=`, `<`, `<=` → Vergleich
  * sonst → einfacher Contains-/Exact-Wert
* **Feld-Whitelist** (unbekanntes Feld → gesamtes Token wird stattdessen als Freitext-Term
  behandelt, kein Fehler):
  * `artist`, `album`, `genre`, `playlist` — Text, Liste=ODER
  * `bpm`/`tempo` (Alias) → `audio_features.tempo`, `energy` → `audio_features.energy`,
    `popularity` → `tracks.popularity` — alle mit Range/Vergleich
  * `year`/`release` (Alias) → `albums.release_date` — Range/Vergleich auf Jahr bzw. Datum
* **Query-Aufbau — zentrale Design-Entscheidung:** jeder Feld-Token liefert über einen eigenen
  Scope eine *Relation, die nur die zu diesem Token passenden Tracks enthält*. Diese wird für
  Join-basierte Felder (`artist`, `playlist` — beides m:n) als **Subquery** eingebunden
  (`Track.where(id: <field_relation>.select(:id))`), nicht als direkter `joins(...).where(...)`
  auf der Hauptrelation. Grund: nur so lässt sich dieselbe Assoziation mehrfach mit
  unterschiedlichen Werten UND-verknüpfen (zwei `playlist:`-Tokens würden bei einem gemeinsamen
  `joins(:playlists)` auf derselben gejointen Zeile stehen und könnten nie gleichzeitig zwei
  verschiedene Namen erfüllen — mit unabhängigen Subqueries pro Token ist das kein Problem, jede
  Subquery liefert unabhängig ihre passenden Track-IDs, die äussere Verkettung schneidet sie).
  Einfache Spalten-Felder (`genre`, `popularity`, `bpm`/`tempo`/`energy` via `json_extract`,
  `album`/`year` via `joins(:album)`) können ohne Subquery direkt verkettet werden, da sie pro
  Track nur einen Wert haben.
  Negation (`-feld:wert`) nutzt dieselbe Feld-Relation, aber `where.not(id: ...select(:id))` statt
  `where(id: ...)` — ein Mechanismus für Position und Negation.
  Freitext-Anteile eines Query-Strings werden weiterhin über das bestehende `Track.search`
  behandelt (Wiederverwendung, kein Duplikat).
* **`Track.search_query(query)`:** neuer Einstiegspunkt, ersetzt den bisherigen Aufruf von
  `Track.search` in `TracksController#index`. Leerer/blanker Query → unveränderte Relation (wie
  bisher).
* **View:** bestehendes Suchfeld bleibt (kein neues Formularfeld), Platzhaltertext/kleiner
  Hilfetext mit 2-3 Beispielen für die Syntax.
* **Autocomplete:** neuer JSON-Endpoint (z.B. `GET /tracks/query_suggestions`), der abhängig vom
  zuletzt eingegebenen Teil-Token entweder passende Feldnamen (statische Liste) oder passende
  Werte vorschlägt — Genres via `Track.distinct.where.not(genre: [nil, ""]).where("genre LIKE ?",
  ...)`, Künstler via `Artist`, Playlists via `Playlist`, jeweils `limit`iert. Kleiner
  Stimulus-Controller am bestehenden Suchfeld (debounced Fetch, Dropdown, Klick fügt Vorschlag ein)
  — kein neues JS-Framework, analog zu bestehenden Stimulus-Controllern im Projekt.
* **Specs:** Parser komplett ohne DB testbar (reine Unit-Tests). Scope-Kombinationen in
  `spec/models/track_spec.rb` (echte DB-Objekte, keine Mocks). Controller-Integration in
  `spec/requests/tracks_spec.rb`. Autocomplete-Endpoint eigener Request-Spec.
