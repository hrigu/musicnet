# 54 - Feature (Anzeige): Aktive Kategorie (Alle/Blues/Fusion) als Anzeige-Filter

## 1. Why

### 1.1 Objective

Der DJ legt mal an Blues-, mal an Fusion-Tanzanlässen auf und möchte über eine Einstellung
steuern können, dass Tracks, Playlists, Artists und die Suche **nur** die gerade relevante
Kategorie zeigen — für einen besseren Überblick, ohne durch die jeweils andere Kategorie
abgelenkt zu werden.

### 1.2 Context

**Bewusst getrennt von der Spotify-Sync-Logik** (Korrektur gegenüber einem ersten Entwurf, siehe
Chat-Diskussion): `SpotifyPlaylistsGateway#owned_fusion_or_blues_playlist?`
(`app/services/spotify_playlists_gateway.rb:118-120`, hartkodiertes `/fusion|blues/i`) bestimmt,
was überhaupt von Spotify importiert wird — das bleibt in diesem Intent **unverändert**. Der DJ
hat sehr viele Playlists auf Spotify; eine konfigurierbare Import-Auswahl (welche Playlist-Muster
überhaupt importiert werden) ist eine separate, spätere Idee und **nicht** Teil dieses Intents.

Dieser Intent führt stattdessen einen reinen **Anzeige-Filter** ein: eine pro Nutzer
gespeicherte "aktive Kategorie" (Alle/Blues/Fusion), die einschränkt, was auf `/tracks`,
`/playlists`, `/artists` und in der DSL-Suche überhaupt sichtbar ist — unabhängig davon, was
lokal in der DB liegt oder importiert wurde. Bewusst als eigenständiges, leicht erweiterbares
Konzept angelegt (weitere Kategorien liessen sich später ergänzen), getrennt von der
Import-Frage.

### 1.3 Depends On Intents

* 1 - Feature (Spotify-Playlists): Sync from Spotify — liefert die Playlist-Namenskonvention
  (Fusion/Blues im Namen), auf der dieser Anzeige-Filter aufbaut.

## 2.0 What

```gherkin
Feature: Aktive Kategorie als Anzeige-Filter

  Als DJ
  möchte ich einstellen können, ob ich nur Blues-, nur Fusion-, oder alle Tracks/Playlists sehe
  damit ich je nach Anlass einen besseren Überblick habe.

  Scenario: Standardverhalten unverändert (Alle)
    Given die Einstellung steht auf "Alle" (Standard)
    When ich /tracks, /playlists oder /artists öffne, oder suche
    Then sehe ich wie bisher alles, unabhängig von der Kategorie

  Scenario: Nur Blues
    Given die Einstellung steht auf "Nur Blues"
    When ich /tracks öffne
    Then sehe ich nur Tracks, die in mindestens einer Blues-Playlist vorkommen
    When ich /playlists öffne
    Then sehe ich nur Blues-Playlists
    When ich /artists öffne
    Then sehe ich nur Künstler mit mindestens einem Track in einer Blues-Playlist
    When ich einen Suchbegriff eingebe (auch mit der DSL)
    Then finde ich nur Treffer aus Blues-Playlists

  Scenario: Nur Fusion
    Given die Einstellung steht auf "Nur Fusion"
    When ich /tracks, /playlists, /artists öffne oder suche
    Then sehe ich nur Fusion-bezogene Treffer, analog zu "Nur Blues"

  Scenario: Import bleibt unberührt
    Given die Einstellung steht auf "Nur Blues"
    When ich "Fetch all Playlists!" auslöse
    Then werden weiterhin sowohl Fusion- als auch Blues-Playlists importiert/aktualisiert
    (der Anzeige-Filter wirkt sich nicht auf den Sync aus)

  Scenario: Einstellung über eine eigene Seite änderbar
    Given ich bin auf der Einstellungen-Seite
    When ich die Kategorie ändere und speichere
    Then wirkt sich das sofort auf Tracks/Playlists/Artists/Suche aus
```

## 3.0 How

### 3.1 Implementation Context

* **Migration:** `add_column :users, :active_playlist_category, :string, default: "all",
  null: false`.
* **`User`:** `ACTIVE_PLAYLIST_CATEGORIES = %w[all blues fusion].freeze`,
  `validates :active_playlist_category, inclusion: { in: ACTIVE_PLAYLIST_CATEGORIES }`,
  `CATEGORY_NAME_SUBSTRINGS = { "blues" => "blues", "fusion" => "fusion" }.freeze` (bewusst
  einfache Teilstring-Suche statt Regex — SQLite kennt kein `REGEXP` ohne Erweiterung, und ein
  Teilstring-Vergleich reicht für "blues"/"fusion" im Playlist-Namen völlig), `#active_category_substring`
  liefert `nil` für `"all"` (oder einen unbekannten Wert) — `nil` bedeutet "kein Filter" für die
  Scopes unten.
* **Neue Model-Scopes** `in_active_category(substring)`, `nil`/leer → unveränderte Relation:
  * `Track.in_active_category` — Subquery über `joins(:playlists)` (gleiches Pattern wie
    `by_artist`/`by_playlist` aus Intent 43, vermeidet Join-Fanout-Probleme in Kombination mit
    Suche/Sortierung).
  * `Playlist.in_active_category` — direkter `LOWER(name) LIKE ?`-Vergleich.
  * `Artist.in_active_category` — Subquery über `joins(tracks: :playlists)`.
* **Controller-Wiring:** `TracksController#index`, `PlaylistsController#index`,
  `ArtistsController#index` verketten `.in_active_category(current_user.active_category_substring)`
  an ihre bestehenden Relationen — bei `TracksController#index` **nach** `search_query`, damit
  auch die DSL-Suche den Filter respektiert (eine `.where(...)`-Verkettung nach einem intern per
  `.or()` gebildeten Suchergebnis wendet sich korrekt auf das gesamte ODER-Ergebnis an, siehe
  Intent 47).
* **`SettingsController`** (neu, `resource :settings, only: %i[edit update]`): `#edit` zeigt ein
  Formular mit den drei Optionen, `#update` speichert `current_user.active_playlist_category`.
* **Navbar:** neuer Link "Einstellungen" im rechten Bereich (bei Hilfe/Avatar/Logout).
* **Spotify-Sync bleibt unverändert** — kein Code in `SpotifyPlaylistsGateway`/
  `BuildMusicNetService` wird in diesem Intent angefasst.
* **Specs:** `User#active_category_substring` (drei Fälle + Fallback bei unbekanntem Wert),
  je ein Scope-Spec für `Track`/`Playlist`/`Artist`, Request-Specs für die drei Controller
  (inkl. Kombination mit bestehender Suche/Sortierung bei Tracks) und für
  `SettingsController`.
