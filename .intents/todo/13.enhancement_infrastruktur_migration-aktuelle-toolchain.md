# 13 - Enhancement (Infrastruktur): Migration auf aktuelle Toolchain

## Why

Ruby 3.2.2, Rails-Gem 7.1.1 (Framework-Defaults sogar nur auf Stand 6.1 via
`config.load_defaults 6.1`), Devise 4.x, omniauth-spotify 0.0.13, sqlite3-Gem 1.6.7 — alles
mehrere Major-Versionen hinter dem aktuellen Stand. Rails 7.1 ist inzwischen aus dem
Security-Support gefallen. Je länger gewartet wird, desto grösser der Sprung und das Risiko.

## Depends On Intents

* 12 - Enhancement (Infrastruktur): Altlasten aufräumen vor der Migration (sollte zuerst
  abgeschlossen sein, damit der Migrations-Diff sauber bleibt)

## What

Ziel-Stack: aktuelles Ruby 3.4.x/3.5.x (genaue Patch-Version zum Zeitpunkt der Umsetzung prüfen,
lokale `rbenv`-Definitionsliste war beim Verfassen dieses Intents veraltet), Rails 8.1.x,
`load_defaults 8.1`, Devise 5.x, omniauth-spotify 1.0.x, sqlite3-Gem 2.x, Rack 3.

Stand aus `bundle outdated` (Zeitpunkt der Erstellung dieses Intents):

| Gem | Aktuell | Ziel | Risiko |
|---|---|---|---|
| ruby | 3.2.2 | 3.4.x/3.5.x | niedrig |
| rails | 7.1.1 | 8.1.3 | hoch (Major, Rack 2→3) |
| load_defaults | 6.1 (→ 7.1 in Intent 12) | 8.1 | hoch (Verhaltensänderungen) |
| sqlite3 (gem) | 1.6.7 | 2.9.5 | mittel (Major, API-Änderungen) |
| devise | 4.9.3 | 5.0.4 | mittel (Major) |
| omniauth-spotify | 0.0.13 | 1.0.3 | hoch (viele Majors auf einmal, kleines/wenig gepflegtes Gem) |
| omniauth-rails_csrf_protection | 1.0.1 | 2.0.1 | mittel |
| rspec-rails | 6.0.3 | 8.0.4 | niedrig (nur Testtooling) |
| turbo-rails/stimulus-rails/importmap-rails | je 1.x | 2.x | mittel |

## Constraints

Kein CI vorhanden — jede Phase braucht manuelle Verifikation (inkl. echtem Spotify-OAuth-Login,
`spotdl`-Download, Mixxx-Export) zusätzlich zur Testsuite. Einzelbetrieb (kein
produktionskritischer Multi-User-Service), daher sequenziell/vorsichtig statt big-bang.

## How

Jede Phase (2.0-8.0) wird bei Umsetzung als eigener, kleinerer Intent ausgeplant. Hier nur die
Reihenfolge/Roadmap:

* [ ] 1.0 - Sicherheitsnetz prüfen: bestehende Testsuite (47 Tests, Intents 7-11) muss vor jeder
      Phase grün sein und bleiben — Migrationsfortschritt wird daran gemessen
* [ ] 2.0 - Phase A (niedriges Risiko, unabhängig): Ruby auf aktuelle Patch-Version anheben,
      `bundle update` für reine Testtooling-/Dev-Gems (rspec-rails, rubocop, byebug, web-console,
      listen). `load_defaults` 6.1→7.1 ist bereits Teil von Intent 12 (Voraussetzung), nicht mehr
      hier
* [ ] 3.0 - Phase B: sqlite3-Gem 1.x → 2.x (Kompatibilität mit aktueller Rails-Version zuerst
      prüfen)
* [ ] 4.0 - Phase C: Devise 4 → 5 (Changelog auf Breaking Changes prüfen, v.a.
      Warden/Response-Format)
* [ ] 5.0 - Phase D: omniauth-spotify 0.0.13 → 1.0.x (höchstes Risiko — vorher Changelog/Diff des
      Gems lesen; danach echten Spotify-Login manuell verifizieren, nicht nur den gemockten Test)
* [ ] 6.0 - Phase E: Rails 7.1 → 8.x (grösster Schritt: Rack 2→3, `load_defaults 7.1→8.1`,
      importmap/turbo/stimulus im Gleichschritt aktualisieren, `bin/rails server`-Startverhalten
      prüfen)
* [ ] 7.0 - Phase F: Aufräumen — verbleibende Deprecation-Warnungen beseitigen,
      `.ruby-version`/Gemfile/README aktualisieren
