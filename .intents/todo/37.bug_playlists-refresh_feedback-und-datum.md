# 37 - Bug/Enhancement (Playlists-Refresh): Verlässliches Feedback + "zuletzt aktualisiert"-Datum

## 1. Why

### 1.1 Objective

Beim Klick auf "Playlist aktualisieren" soll der DJ zuverlässig sehen, was sich geändert hat
(oder dass nichts geändert hat), und die Playlist-Seite soll immer anzeigen, wann sie zuletzt
synchronisiert wurde.

### 1.2 Context

Analyse des `development.log` zeigt: seit der GET→POST-Umstellung (wegen Turbo-8-Hover-Prefetch,
diese Session) sendet der Browser den Refresh-Request jetzt als `POST .../refresh` mit
`Accept: text/vnd.turbo-stream.html, ...`. `PlaylistsController#refresh` rendert bei Erfolg direkt
`:show` (200), statt wie `download`/`fetch_all` bei Erfolg zu redirecten — genau die von
Rails/Turbo empfohlene Praxis ("immer nach einer nicht-GET-Aktion redirecten") wird hier verletzt.
Das ist die wahrscheinlichste Ursache für das fehlende Feedback und ein Turbo-Anti-Pattern,
unabhängig vom genauen Symptom.

Zusätzlich: `updated_at` wird bereits heute bei jedem `refresh_playlist`-Aufruf zuverlässig
gesetzt (`sync_playlist_with_spotify` ruft `playlist.update!(...)` immer auf, unabhängig vom
Snapshot-Vergleich) und beim Bulk-Sync (`fetch_all`) nur, wenn sich tatsächlich etwas geändert
hat — passt inhaltlich schon gut zu "zuletzt aktualisiert". Kleine Ungenauigkeit: ein manuelles
Umbenennen über `edit`/`update` würde `updated_at` ebenfalls anfassen. Für ein reines
Anzeige-Feature akzeptiert (Nutzerentscheidung) — kein neues Feld, keine Migration nötig.

### 1.3 Related to Intents

* 34 - Feature (Tracks-Index) / diese Session — GET→POST-Umstellung, die den Bug ausgelöst hat.
* 31 - Enhancement (Spotify-Playlists): Snapshot-Sync — Grund, warum `updated_at` beim Bulk-Sync
  nur bei echten Änderungen gesetzt wird (bewusst beibehalten, nicht Teil dieses Intents).

## 2.0 What

```gherkin
Feature: Playlist-Refresh-Feedback und Sync-Datum

  Als DJ
  moechte ich nach "Playlist aktualisieren" zuverlaessig sehen was sich geaendert hat,
  und immer sehen wann die Playlist zuletzt synchronisiert wurde.

  Scenario: Refresh mit Aenderungen
    Given eine Playlist, bei der sich Tracks geaendert haben
    When ich auf "Playlist aktualisieren" klicke
    Then lande ich wieder auf der Playlist-Seite
    And sehe eine Meldung mit hinzugefuegten und entfernten Tracks

  Scenario: Refresh ohne Aenderungen
    Given eine Playlist ohne Aenderungen auf Spotify
    When ich auf "Playlist aktualisieren" klicke
    Then sehe ich die Meldung "Keine Aenderungen - die Playlist ist auf dem aktuellen Stand"

  Scenario: Zuletzt-aktualisiert-Datum immer sichtbar
    Given eine beliebige Playlist-Seite
    When ich sie oeffne
    Then sehe ich das Datum/Uhrzeit von updated_at der Playlist
```

## 3.0 How

### 3.1 Implementation Context

* `PlaylistsController#refresh`: Erfolgsfall auf `redirect_to playlist_path(@playlist)` umstellen
  (analog `download`), Ergebnis (`RefreshInfo#added`/`#removed`) über
  `flash[:refresh_added]`/`flash[:refresh_removed]` transportieren statt über `@refresh_info`.
* `show.html.erb`: bestehenden Alert-Block auf `flash[:refresh_added]`/`flash[:refresh_removed]`
  umstellen (Anwesenheit der Flash-Keys statt `@refresh_info` prüfen); zusätzlich immer
  `<%= l(@playlist.updated_at, format: :long) %>` (o. ä. Locale-Format) neben dem Titel anzeigen.
* Bestehende Request-Specs für `POST /playlists/:id/refresh` müssen auf `redirect_to` +
  `follow_redirect!` + Flash-Check umgestellt werden.
