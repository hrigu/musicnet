# 28 - Enhancement (Tracks-Track): Genre in der DB cachen

## 1. Why

### 1.1 Objective

`Track#genre` soll das Genre aus einer DB-Spalte liefern statt bei jedem Seitenaufbau die
Audio-Datei mit WahWah zu parsen — das spart auf der Track-Übersicht die verbleibenden ~1.3s
Datei-Lesen (2331 Dateien pro Aufruf).

### 1.2 Context

Nach Intent 27 rendert `/tracks` in ~3.7s statt 23s. Grösster verbleibender Posten ist das
Genre: Es steht nur in den Metadaten der heruntergeladenen `.m4a`-Dateien und wird pro Zeile
via `WahWah.open` gelesen (~0.55ms pro Datei). Das Genre einer Datei ändert sich praktisch
nie — ein klassischer Cache-Fall.

### 1.3 Depends On Intents

* 27 - Enhancement (Tracks-Track): Index-Seite ohne Query- und Glob-Lawine rendern
  (memoisiertes `track_path` hält den Datei-Check billig)

## 2.0 What

```gherkin
Feature: Tracks Track - Genre in der DB cachen

  Als DJ
  möchte ich dass das Genre nach dem ersten Lesen in der DB liegt
  damit die Track-Übersicht nicht bei jedem Aufruf tausende Dateien öffnet.

  Scenario: Genre wird beim ersten Lesen gecacht
    Given ein Track mit heruntergeladener Datei inkl. Genre-Tag
    When ich das Genre zum ersten Mal lese
    Then wird es aus der Datei gelesen und in der DB gespeichert
    And beim nächsten Lesen wird die Datei nicht mehr geöffnet

  Scenario: Genre ohne Datei
    Given ein Track ohne heruntergeladene Datei
    When ich das Genre lese
    Then ist es nil und es wird nichts in der DB gespeichert
```

## 3.0 How

### 3.1 Implementation Context

Read-Through-Cache in `Track#genre`:

1. Migration `AddGenreToTracks`: String-Spalte `genre` auf `tracks`, nullable
   (Timestamp mit Sekunden-Präzision).
2. `Track#genre` liefert `self[:genre]`, wenn vorhanden. Sonst wie bisher via
   `track_path`/`WahWah` lesen und — nur bei persistiertem Track und vorhandenem Wert —
   per `update_column` speichern (kein Validierungs-/Callback-Lauf, `updated_at` bleibt
   unangetastet, es ist nur ein Cache).

Der erste `/tracks`-Aufruf nach der Migration füllt den Cache automatisch (einmalig ~1.3s),
danach fallen die Datei-Reads weg. Kein Backfill-Task nötig.

### 3.2 Design-Entscheidungen

* Nur vorhandene Genres werden gecacht: Dateien ohne Genre-Tag werden weiterhin pro Aufruf
  gelesen (typischerweise wenige, je ~0.55ms) — dafür gibt es kein Stale-Leer-Problem, wenn
  eine Datei später getaggt wird.
* Invalidierung bewusst manuell: Wird eine Datei neu heruntergeladen/umgetaggt, kann das
  gecachte Genre veralten. Reset bei Bedarf in der Konsole via `Track.update_all(genre: nil)`.
  Kein Automatismus in den Download-Services (Aufwand/Nutzen).
