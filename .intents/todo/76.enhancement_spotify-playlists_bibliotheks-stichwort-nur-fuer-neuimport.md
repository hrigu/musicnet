# 76 - Enhancement (Spotify-Playlists): Bibliotheks-Stichwort nur für Neuimport, nicht für Bestandsschutz

## 1. Why

### 1.1 Objective

Eine bereits in Musicnet vorhandene Playlist soll durch einen Sync nie mehr verschwinden, nur weil ihr
aktueller Name kein Bibliotheks-Stichwort (mehr) enthält. Das Stichwort-Filter soll ausschliesslich
entscheiden, welche komplett neuen, bisher unbekannten Spotify-Playlists importiert werden - nicht, ob
eine bereits bekannte Playlist erhalten bleibt.

### 1.2 Context

Beim Planen einer künftigen "Neue Playlist anlegen"-Funktion (Push zu Spotify) fiel ein bestehender
Architektur-Fallstrick auf: `SpotifyPlaylistsGateway#all` (genutzt von `BuildMusicNetService#build`, dem
vollen Sync) filtert aktuell auf Playlists, die **sowohl** dem aktuellen User gehören **als auch** ein
Bibliotheks-Stichwort im Namen tragen (`owned_library_playlist?`). Diese gefilterte Liste steuert nicht
nur, welche neuen Playlists importiert werden, sondern auch `delete_vanished_playlists` - jede lokale
Playlist, deren `spotify_id` nicht in dieser Liste auftaucht, wird als "auf Spotify verschwunden"
behandelt und gelöscht.

Das bedeutet konkret: Eine bereits in Musicnet vorhandene Playlist, deren Name (durch Umbenennung auf
Spotify oder künftig direkt in Musicnet, siehe Intent 75) kein Bibliotheks-Stichwort mehr enthält, würde
beim nächsten vollen Sync lokal gelöscht - **obwohl sie auf Spotify weiterhin existiert**. Eine künftig
direkt in Musicnet neu angelegte Playlist wäre davon genauso betroffen, sollte ihr Name zufällig kein
Stichwort treffen.

**Recherche-Ergebnis - kein Performance-Nachteil:** `SpotifyPlaylistsGateway#all` ruft `fetch_pages
(:playlists)` auf, was ohnehin bereits **alle** Playlists des Users von Spotify holt (Spotify unterstützt
keine serverseitige Namens-Filterung) - der bisherige `.select { owned_library_playlist? }` ist ein rein
lokaler, kostenloser Ruby-Filter über bereits geladene Metadaten. Die teuren Operationen (`tracks_for`,
`prefetch_details`) laufen schon heute nur für Playlists, die tatsächlich neu angelegt werden oder deren
`snapshot_id` sich geändert hat - das bleibt unverändert. `refresh_playlist` (Einzel-Refresh) ist bereits
unbetroffen: `SpotifyPlaylistsGateway#find` sucht ohnehin ungefiltert über alle Playlists des Users.

## 2.0 What

```gherkin
Feature: Bibliotheks-Stichwort nur fuer Neuimport

  Als DJ
  moechte ich, dass eine bereits in Musicnet vorhandene Playlist beim Sync erhalten bleibt,
  auch wenn ihr Name kein Bibliotheks-Stichwort (mehr) enthaelt
  damit ich Playlists umbenennen oder neu anlegen kann, ohne sie beim naechsten Sync zu verlieren.

  Scenario: Neue, nur auf Spotify existierende Playlist ohne passendes Stichwort
    Given eine Playlist auf Spotify, die noch nicht in Musicnet existiert
    And ihr Name enthaelt kein Stichwort einer konfigurierten Library
    When ein voller Sync laeuft
    Then wird diese Playlist nicht importiert

  Scenario: Neue, nur auf Spotify existierende Playlist mit passendem Stichwort
    Given eine Playlist auf Spotify, die noch nicht in Musicnet existiert
    And ihr Name enthaelt das Stichwort einer konfigurierten Library
    When ein voller Sync laeuft
    Then wird diese Playlist importiert und der Library zugeordnet

  Scenario: Bereits lokal vorhandene Playlist verliert ihr passendes Stichwort
    Given eine Playlist existiert bereits lokal in Musicnet
    And sie wird auf Spotify so umbenannt, dass ihr Name kein Stichwort mehr enthaelt
    When ein voller Sync laeuft
    Then bleibt die Playlist lokal erhalten
    And ihr Name und ihre Tracks werden trotzdem mit Spotify abgeglichen
    And ihre Bibliothekszuordnung wird leer, da kein Stichwort mehr passt

  Scenario: Playlist wird auf Spotify wirklich entfernt/entfolgt
    Given eine Playlist existiert bereits lokal in Musicnet
    When sie auf Spotify nicht mehr in der eigenen Playlist-Liste auftaucht
    Then wird sie lokal als verwaist geloescht (unveraendertes Verhalten)
```

## 3.0 How

### 3.1 Implementation Context

* **`SpotifyPlaylistsGateway#all`** (`app/services/spotify_playlists_gateway.rb`): filtert nur noch auf
  Besitzer (`playlist.owner.id == @spotify_user.id`), kein `Library.matching` mehr. Die private Methode
  `owned_library_playlist?` wird entfernt (ihr einziger verbleibender Bestandteil, der Owner-Check,
  wandert direkt in `all`).
* **`BuildMusicNetService#build`** (`app/services/build_music_net_service.rb`): Das Stichwort-Gate
  (`Library.matching(spot_playlist.name).any?`) wandert in die `if local_playlist.nil?`-Verzweigung (gilt
  nur für den Neuimport-Entscheid). Der `elsif local_playlist.snapshot_id != spot_playlist.snapshot_id`-
  Zweig (Abgleich einer bereits bekannten Playlist) bleibt ungated - er läuft für jede lokal bekannte
  Playlist, solange sie noch unter `own_playlists` (jetzt: alle eigenen Spotify-Playlists) auftaucht.
  `delete_vanished_playlists(own_playlists.map(&:id))` bekommt dadurch automatisch die vollständige,
  ungefilterte Liste als Bestandsschutz-Grundlage.
* `assign_libraries` (Bibliothekszuordnung) bleibt unverändert: eine Playlist ohne passendes Stichwort
  bekommt einfach eine leere `libraries`-Zuordnung (kein Sonderfall nötig, bestehende Logik).
* **Tests**: `spec/services/spotify_playlists_gateway_spec.rb` `#all` - Erwartung ändert sich (liefert
  jetzt auch namentlich nicht-passende, aber eigene Playlists mit); der Library-spezifische Testfall
  ("berücksichtigt frei konfigurierte Libraries") wandert nach `build_music_net_service_spec.rb`, da die
  Stichwort-Logik jetzt dort lebt. Bestehender Test "wenn eine eigene Playlist weder 'fusion' noch
  'blues' im Namen hat" (Neuimport-Gate) bleibt inhaltlich gültig. Neuer Test: eine bereits lokal
  existierende Playlist, die ihr Stichwort verliert (Rename via Spotify), bleibt beim nächsten Sync
  erhalten, wird weiter aktualisiert (Name/Tracks/snapshot_id) und ihre `libraries`-Zuordnung wird leer.
* **`CLAUDE.md`**: Abschnitt "Bibliotheken (Libraries)" - "Import filter"-Beschreibung aktualisieren
  (Stichwort gilt nur für Neuimport, nicht für Bestandsschutz/Löschung).
