# 90 - Feature (Tracks-Tags): Neues Tag im Inline-Widget anlegen

## 1. Why

Im kompakten "+"-Zuweisen-Widget (Intent 83, jetzt auf `/tracks`, `/artists/:id` und
`/playlists/:id` verfügbar, Intent 89) lässt sich bisher nur ein **bestehendes** Tag zuweisen -
findet die Livesuche keinen Treffer, gibt es keine Möglichkeit, direkt dort ein neues Tag
anzulegen. Der DJ muss dafür auf die Track-Detailseite wechseln, die diesen Schritt bereits kennt
(Intent 79: Name eingeben -> Kategorie wählen -> Stärke). Das kehrt eine bewusste Einschränkung aus
Intent 83 um ("kein 'Neuer Tag'-Zweig ... neue Tags anlegen bleibt der Track-Detailseite
vorbehalten") - der DJ möchte das jetzt auch inline.

### 1.1 Depends On Intents

* 83 - Feature: das Inline-Widget selbst (`tag_assign_controller.js`, `tracks/_tag_assign_inline.erb`)
* 89 - Feature: macht das Widget auf drei Seiten gleichzeitig verfügbar

## 2.0 What

```gherkin
Feature: Neues Tag im Inline-Widget anlegen

  Als DJ
  moechte ich, wenn die Livesuche im Inline-Widget kein passendes Tag findet, direkt dort ein
  neues anlegen koennen
  damit ich dafuer nicht auf die Track-Detailseite wechseln muss.

  Scenario: Neues Tag ueber das Inline-Widget anlegen
    Given ich habe im Inline-Widget (z.B. auf /tracks) einen Namen getippt, der noch kein Tag ist
    When ich "Neuer Tag: „…“" waehle und eine Kategorie bestaetige
    Then wird ein neues Tag in dieser Kategorie angelegt und dem Track zugewiesen (Standard-Staerke,
    kein zusaetzlicher Staerke-Schritt - konsistent mit der Sofort-Zuweisung eines bestehenden Tags
    in diesem Widget)

  Scenario: Bestehendes Verhalten bleibt unveraendert
    Given ich waehle im Inline-Widget einen bestehenden Tag-Treffer
    Then wird er weiterhin sofort mit Standard-Staerke zugewiesen (kein Kategorie-/Staerke-Schritt)
```

## 3.0 How

### 3.1 Implementation Context

* **`tag_assign_controller.js`:** `renderResults` zeigt "Neuer Tag: „…“" jetzt in jedem Modus, nicht
  nur wenn `existingOnlyValue` falsch ist. `confirmCategory` verzweigt: im `existingOnly`-Modus wird
  nach der Kategorie-Wahl sofort abgeschickt (`panelTarget.requestSubmit()`, gleiche
  Standard-Staerke wie beim Zuweisen eines bestehenden Tags), sonst wie bisher zum Staerke-Schritt.
  `backToSearch` greift defensiv auf `stepStrengthTarget` zu (`hasStepStrengthTarget`), da das
  Inline-Widget dieses Ziel nicht rendert.
* **`tracks/_tag_assign_inline.erb`:** ergaenzt einen `stepCategory`-Block (Kategorie-Auswahl +
  "Weiter"/"Zurück"), analog zum vollen Formular auf der Detailseite - ohne eigenen
  Staerke-Schritt.
* **Bestehender Test widerspricht dem neuen Verhalten:** `spec/system/tag_assign_inline_spec.rb`
  ("bietet keine Möglichkeit, hier ein neues Tag anzulegen") wird durch einen Test ersetzt, der das
  neue Verhalten prüft.
