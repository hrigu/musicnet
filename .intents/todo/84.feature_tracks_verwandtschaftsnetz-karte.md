# 84 - Feature (Tracks): Verwandtschaftsnetz-Karte

## 1. Why

### 1.1 Objective

Vision des DJs: Tracks als Punkte auf einer Karte ("Map") darstellen, wobei die räumliche Nähe
bzw. sichtbare Verbindungen zwischen Punkten die "Verwandtschaft" zweier Tracks ausdrücken -
abgeleitet aus den gemeinsamen Tags und deren Stärke (`TrackTag#strength`, Intent 77/79/81/83).
Ziel: verwandte Tracks visuell entdecken statt nur über Tabellen/Suche zu finden - z.B. um beim
Auflegen einen ähnlichen Anschlusstrack zu einem gerade laufenden Track zu finden.

### 1.2 Context

**Dieses Intent ist ausdrücklich noch nicht umsetzungsreif.** Der DJ hat die Idee bewusst zur
weiteren Klärung angemeldet ("Diese Idee müssen wir noch etwas genauer anschauen"), bevor eine
Aufgabenliste (Abschnitt 3.2) sinnvoll definiert werden kann. Dieses Dokument hält die Vision und
die bekannten offenen Fragen fest, damit sie nicht verloren geht, nicht aber ein fertiges Konzept.

### 1.3 Depends On Intents

* 77 - Feature (Tags): liefert `Category`/`Tag`/`TrackTag` inkl. `strength` - die Datengrundlage
  für jede Verwandtschafts-Berechnung.
* 82 - Feature (Categories): für die Neuzuordnung ausgeblendete Kategorien - zu klären, ob deren
  Tags auch aus der Verwandtschafts-Berechnung ausgeschlossen werden sollen oder nicht (siehe
  offene Fragen unten).

## 2.0 What (vorläufig, noch nicht final)

```gherkin
Feature: Verwandtschaftsnetz-Karte

  Als DJ
  möchte ich meine Tracks als Punkte auf einer Karte sehen, die anhand gemeinsamer Tags und deren
  Stärke räumlich/visuell verbunden sind
  damit ich verwandte Tracks (z.B. für einen guten Anschlusstrack) visuell entdecken kann, statt
  nur über Tabellen und Textsuche zu suchen.

  Scenario: Grobe Vision (Details offen, siehe CONSTRAINTS)
    Given zwei Tracks teilen sich mehrere Tags mit hoher Stärke
    Then erscheinen sie auf der Karte nahe beieinander / deutlich verbunden
    And zwei Tracks ohne gemeinsame Tags erscheinen weit auseinander / unverbunden
```

## 3.0 How

### 3.1 Noch zu klärende Fragen, bevor Tasks definiert werden können

* **Verwandtschafts-Metrik:** Wie wird aus zwei Tag-Stärke-Profilen eine einzelne
  "Verwandtschafts"-Zahl? Optionen z.B. Summe der Minima gemeinsamer Tag-Stärken, Kosinus-
  Ähnlichkeit der Stärke-Vektoren, oder etwas Einfacheres (Anzahl gemeinsamer Tags gewichtet nach
  Stärke). Muss mit dem DJ anhand konkreter Beispieltracks durchgespielt werden.
* **Positionierung auf der Karte:** Verwandtschaft ist eine Beziehung zwischen Paaren, eine 2D-
  Karte kann aber nicht beliebig viele paarweise Abstände gleichzeitig exakt abbilden. Braucht es
  eine Dimensionsreduktion (z.B. eine Kraft-basierte Graph-Layout-Bibliothek, die Tracks mit
  starker Verwandtschaft zueinander zieht), oder reicht eine reine Graph-Darstellung (Knoten +
  Kanten, keine echte "Karte" mit fester Position)?
* **Kanten vs. Nähe:** zeigt die Karte explizite Verbindungslinien (mit z.B. Dicke = Stärke) oder
  wirkt Verwandtschaft nur indirekt über die Position der Punkte?
* **Skalierung:** wie viele Tracks/Kanten müssen gleichzeitig performant darstellbar sein? Das
  bestimmt z.B. ob eine einfache SVG-Lösung reicht oder Canvas/WebGL nötig ist.
* **Technologie:** dieses Projekt hat bewusst keinen Node/Yarn-Build-Schritt (Hotwire/Stimulus via
  importmap, siehe `CLAUDE.md`). Eine Graph-/Netzwerk-Visualisierungsbibliothek (z.B. eine
  Force-Directed-Graph-Library) müsste importmap-kompatibel sein oder vendored werden - zu prüfen,
  was ohne Build-Schritt sauber einbindbar ist.
* **Einbezug ausgeblendeter Kategorien:** sollen Tags aus für die Neuzuordnung ausgeblendeten
  Kategorien (Intent 82, `Category#hidden_for_assignment`) in die Verwandtschafts-Berechnung
  einfliessen, oder nur "aktive" Kategorien? Das Flag wurde bisher nur für die Zuweisungs-Suche
  entworfen, nicht für Auswertungen wie diese.
* **Umfang/Filter:** komplett Tracks-DB oder gefiltert (z.B. nach aktiver `Library`, Intent 57)?
