# 83 - Feature (Tracks-Tags): Bestehendes Tag inline in der Tracks-Liste zuweisen

## 1. Why

### 1.1 Objective

Ein Tag einem Track zuzuweisen geht bisher nur über die Track-Detailseite (Intent 79). Für den
täglichen Workflow soll das auch direkt aus der `/tracks`-Übersicht möglich sein, analog zum
bereits vorhandenen Inline-Editieren der Stärke (Intent 81): ein kleines "+" neben den Tag-Badges
einer Zeile öffnet eine Suche, ein Klick auf einen Treffer verknüpft sofort.

### 1.2 Context

Bewusst eingeschränkt gegenüber dem vollen Zuweisungs-Flow der Detailseite: hier lassen sich nur
**bestehende** Tags zuweisen, kein "Neuer Tag"-Zweig mit Kategorie-Auswahl - das hat sich in der
Planung als unnötige Komplexität für die Listenansicht herausgestellt, komplett neue Tags anlegen
bleibt Aufgabe der Track-Detailseite. Die Stärke wird nicht erfragt, sondern fix auf 5 gesetzt -
Feintuning läuft über Intent 81 weiter.

### 1.3 Depends On Intents

* 79 - Feature (Tracks): liefert `TagsController#search` (Livesuche) und
  `TrackTagsController#create` (`resolve_tag`/`find_or_initialize_by`), hier wiederverwendet statt
  dupliziert.
* 81 - Feature (Tracks-Tags): liefert das Turbo-Stream-/Wrapper-Muster
  (`components/_track_tag_badge.erb`, `dom_id(track_tag, :tag_badge)`) und die Stärke-Nachjustierung
  für die hier fix gesetzte Stärke 5.
* 82 - Feature (Categories): die hier verwendete Suche blendet automatisch Tags aus für die
  Neuzuordnung ausgeblendeten Kategorien aus, ohne dass dieses Intent das selbst behandeln muss.

## 2.0 What

```gherkin
Feature: Bestehendes Tag inline in der Tracks-Liste zuweisen

  Als DJ
  möchte ich einem Track direkt in der Tracks-Übersicht ein bestehendes Tag zuweisen können
  damit ich dafür nicht auf die Track-Detailseite wechseln muss.

  Scenario: Bestehendes Tag zuweisen
    Given ein Track ohne dieses Tag wird in der Tracks-Tabelle angezeigt
    When ich auf das "+" neben seinen Tags klicke, einen Tag-Namen eintippe und einen Treffer wähle
    Then wird der Track mit diesem Tag und Stärke 5 verknüpft, das Badge erscheint sofort in der Zeile

  Scenario: Schon zugewiesene Tags werden nicht doppelt vorgeschlagen
    Given ein Track trägt bereits ein bestimmtes Tag
    When ich das "+"-Suchfeld für diesen Track öffne und nach diesem Tag suche
    Then erscheint es nicht in den Vorschlägen

  Scenario: Kein Treffer bedeutet keine Möglichkeit, hier ein neues Tag anzulegen
    Given ich tippe einen Namen ohne bestehenden Treffer
    Then gibt es hier keine "Neuer Tag"-Option (anders als auf der Track-Detailseite)

  Scenario: Abbruch
    Given das Such-Widget für ein Tag ist offen
    When ich Escape drücke oder ausserhalb klicke
    Then schliesst es sich ohne eine Zuordnung anzulegen
```

## 3.0 How

### 3.1 Implementation Context

* **`TagsController#search`**: zusätzlicher optionaler Parameter `track_id` - wenn gesetzt, werden
  bereits mit diesem Track verknüpfte Tags aus den Ergebnissen ausgeschlossen
  (`where.not(id: track.tag_ids)`).
* **`TrackTagsController#create`**: `format.turbo_stream`-Zweig ergänzt (bisher nur Redirect für
  die Detailseite) - hängt das neue Badge-Partial (`components/_track_tag_badge.erb`, Intent 81)
  in die Tags-Zelle der Zeile an und schliesst/reset das Such-Widget wieder. Bei einem Fehler
  (z.B. kein Tag gewählt) wird stattdessen nur eine Fehlermeldung im Widget angezeigt, keine
  Weiterleitung.
* **View**: neues, kompaktes Partial (nur Suche, kein Kategorie-/Stärke-Schritt) neben der
  Tag-Badge-Liste in `tracks/_track.erb`; die Tags-Zelle bekommt eine stabile `dom_id`, damit der
  Turbo-Stream dort anhängen kann.
* **Stimulus**: `tag_assign_controller.js` (Intent 79) wird nicht kopiert, sondern um einen
  reduzierten Modus ergänzt (kein Kategorie-/Stärke-Schritt, direkter Submit bei Tag-Auswahl) -
  Wiederverwendung der Such-/Tastatur-Logik statt Duplikat.
