# 44 - Feature (Tracks-Suche): Persistierte Suchen

## 1. Why

### 1.1 Objective

Eine mit der neuen DSL (Intent 43) eingegebene Suche soll unter einem Namen gespeichert und später
per Klick wieder hervorgeholt werden können, ohne den Query-String erneut eintippen zu müssen.

### 1.2 Context

Der `q`-Parameter auf `/tracks` ist nach Intent 43 ein DSL-String (z.B. `genre:jazz,fusion
bpm:80..100 playlist:"Fusion Abende" playlist:"Blues Session"`). Eine "gespeicherte Suche" ist
technisch nichts weiter als Name + dieser String — kein neues Datenmodell für die Suche selbst
nötig, nur eine kleine Ablage dafür.

### 1.3 Depends On Intents

* 43 - Feature (Tracks-Index): DSL-basierte Textsuche — liefert den zu speichernden `q`-String.

## 2.0 What

```gherkin
Feature: Tracks-Suche - Persistierte Suchen

  Als DJ
  möchte ich eine eingegebene Suche unter einem Namen speichern und später wieder laden können
  damit ich wiederkehrende Suchen (z.B. "Jazz 80-100 BPM für Set X") nicht erneut eintippen muss.

  Scenario: Aktuelle Suche speichern
    Given ich habe eine Suche über die DSL eingegeben
    When ich ihr einen Namen gebe und auf "Speichern" klicke
    Then ist die Suche unter diesem Namen gespeichert
    And ich sehe eine Bestätigung

  Scenario: Gespeicherte Suche laden
    Given eine gespeicherte Suche existiert
    When ich auf ihren Namen klicke
    Then wird die Tracks-Liste mit dieser Suche gefiltert angezeigt
    And das Suchfeld zeigt den geladenen Query-String

  Scenario: Gespeicherte Suche löschen
    Given eine gespeicherte Suche existiert
    When ich sie lösche
    Then erscheint sie nicht mehr in der Liste

  Scenario: Name muss eindeutig sein
    Given eine gespeicherte Suche mit Namen "Jazz Set" existiert
    When ich versuche, eine weitere Suche unter demselben Namen zu speichern
    Then sehe ich einen Fehlerhinweis, die neue Suche wird nicht gespeichert

  Scenario: Leerer Name oder leere Suche
    Given das Namensfeld oder das Suchfeld ist leer
    When ich auf "Speichern" klicke
    Then wird nichts gespeichert, ich sehe einen Fehlerhinweis
```

## 3.0 How

### 3.1 Implementation Context

* **Modell `SavedSearch`:** `name:string` (required, unique), `query:string` (required),
  Timestamps. Kein User-Bezug — die App ist Single-User (wie `QueueEntry`, Intent 42).
* **Controller `SavedSearchesController`:** nur `create` und `destroy` (kein `index`/`show` als
  eigene Seite — die Liste wird direkt auf `/tracks` mitgerendert). Beide Actions sind Mutationen
  → POST/DELETE + Redirect-mit-Flash, wie im Projekt etabliert (Turbo-8-GET-Prefetch-Falle,
  CLAUDE.md/Refresh-Flow) — kein inline `render` nach einer mutierenden Action.
* **View-Integration auf `/tracks`:**
  * Kleines Formular "Suche speichern" (Namensfeld + aktueller `q`-Wert als Hidden Field),
    **ausserhalb** von `turbo_frame_tag "tracks"` platziert (analog zum persistenten Audio-Player/
    der Queue, Intent 40–42) — vermeidet Frame-Target-Komplexität für eine seltene Aktion, ein
    normaler Redirect zurück auf `/tracks` reicht.
  * Liste der gespeicherten Suchen (Name als Link auf `tracks_path(q: saved_search.query)`,
    daneben ein Lösch-Button) — ebenfalls ausserhalb des Turbo Frame. Der Link ist reine Navigation
    (GET), keine Mutation — anders als die Turbo-8-Gotcha-Fälle im Projekt, wo GET-Hover-Prefetch
    eine *Nebenwirkung* ausgelöst hätte.
* **Specs:** Modell-Validierungen, Controller-Request-Specs (create/destroy inkl. Fehlerfälle),
  keine Mocks nötig (reine DB-Objekte).
