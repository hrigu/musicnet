# 58 - Bug (Playlists-Fetch-All): Kein visuelles Feedback nach Sync

## 1. Why

### 1.1 Objective

"Alle Playlists holen" (`PlaylistsController#fetch_all`) zeigt nach dem Sync keinerlei sichtbare
Rückmeldung an — der Nutzer weiss nicht, ob überhaupt etwas passiert ist.

### 1.2 Context

`fetch_all` rendert nach einem erfolgreichen Sync implizit die View `fetch_all.html.erb`
("Alles geholt" + rohe `@info.hash`-Ausgabe), statt umzuleiten. Der Link, der die Action auslöst,
ist aber ein `data-turbo-method: :post`-Link (`playlists/index.html.erb`) — genau das Muster, das
CLAUDE.md selbst unter "Sync flow" als Anti-Pattern dokumentiert (Intent 37): ein direktes 200-
Render nach einem Turbo-POST kann das Feedback stillschweigend verschlucken. `refresh` und
`download` wurden dafür bereits auf Redirect-nach-Mutation mit Flash umgestellt (Intent 37/38),
`fetch_all` offenbar nie.

### 1.3 Depends On Intents

* 34/37/38 - Redirect-nach-Mutation-Konvention (bereits etabliert bei `refresh`/`download`)

## 2.0 What

```gherkin
Feature: Playlists Fetch-All - Sichtbares Feedback nach Sync

  Als DJ
  möchte ich nach "Alle Playlists holen" eine sichtbare Bestätigung sehen
  damit ich weiss, dass der Sync gelaufen ist und was sich geändert hat

  Scenario: Erfolgreicher Sync mit Änderungen zeigt eine Zusammenfassung
    Given der Sync legt neue Playlists und Tracks an
    When ich "Alle Playlists holen" ausführe
    Then werde ich auf die Playlists-Seite umgeleitet
    And ich sehe eine Zusammenfassung der Änderungen (Anzahl neuer/gelöschter Playlists/Tracks/etc.)

  Scenario: Sync ohne Änderungen zeigt trotzdem eine Bestätigung
    Given der Sync ändert nichts (alle snapshot_ids unverändert)
    When ich "Alle Playlists holen" ausführe
    Then sehe ich eine Bestätigung, dass der Sync gelaufen ist, ohne Fehlermeldung

  Scenario: Ein bereits laufender Sync zeigt weiterhin die bestehende Fehlermeldung
    Given ein Sync läuft bereits
    When ich "Alle Playlists holen" erneut ausführe
    Then sehe ich die bestehende Alert-Meldung (unverändertes Verhalten)
```

## 3.0 How

### 3.1 Implementation Context

`fetch_all` bekommt dieselbe Redirect-mit-Flash-Struktur wie `refresh`/`download`
(`set_refresh_flash`/`set_download_flash` als Vorbild). `ServiceInfo#hash` hat eine bestehende
Eigenart: Werte aus Einzel-Aufrufen (`add_new_created_playlist` etc., ein Aufruf pro Datensatz)
sind flache Arrays von Namen, Werte aus den Lösch-Pfaden (`delete_vanished_playlists`,
`cleanup_orphan_records`, je ein Aufruf mit der ganzen Namensliste als Wert) landen durch
`ServiceInfo#add`s `<<` als eine zusätzliche Verschachtelungsebene (`[[name1, name2]]`) — für
Anzeigezwecke reicht `.flatten.size`, um in beiden Fällen die korrekte Anzahl zu bekommen, ohne
`ServiceInfo` selbst anzufassen (aus Zeitgründen bewusst nicht Teil dieses Bugfixes: der
zugrundeliegende Inkonsistenz in `ServiceInfo#add` wird nicht bereinigt, nur robust umschifft).

Zusammenfassung als ein einzelner Flash-Satz (z.B. "3 Playlists neu, 1 gelöscht · 42 Tracks neu"),
nur nicht-leere Kategorien, analog zum knappen Stil der bestehenden Flash-Meldungen. Keine
Namenslisten (anders als bei `download`/`refresh`) — ein erster Voll-Import kann tausende Namen
umfassen (siehe Intent 33), das würde das Flash-Cookie-Limit sprengen (`CookieOverflow`, schon
einmal in Intent 38 aufgetreten).

Die jetzt nie mehr gerenderte View `app/views/playlists/fetch_all.html.erb` wird entfernt.
