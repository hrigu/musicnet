# 60 - Bug (Downloads): Unlesbare spotdl-Ausgabe

## 1. Why

### 1.1 Objective

Beim Herunterladen von Tracks erzeugt `spotdl` eine riesige, kaum lesbare Wand aus Text statt
einer kompakten Anzeige (aktueller Song, Erfolg/Misserfolg).

### 1.2 Context

`DownloadPlaylistService#download` ruft `spotdl` über `system(builder.build, chdir: tracks_dir)`
auf, ohne stdout/stderr umzuleiten — der spotdl-Prozess erbt direkt das stdout des Rails-
Prozesses. spotdl rendert standardmässig einen animierten Rich-Fortschrittsbalken, der eine echte
interaktive TTY mit In-Place-Redraw voraussetzt (Cursor-Escape-Codes). Landet die Ausgabe
stattdessen in einem Log/einer nicht-interaktiven Pipe, kann kein In-Place-Redraw stattfinden —
jeder einzelne Redraw-Tick wird stattdessen als kompletter neuer Textblock angehängt, viele Male
pro Song.

spotdl bietet dafür bereits eine fertige Lösung: `--simple-tui` schaltet auf eine einfache,
nicht-animierte Text-Ausgabe um (eine Zeile pro Ereignis statt eines redrawing Balkens) — exakt
das gewünschte Verhalten (Song, das gerade läuft; Erfolg/Misserfolg), ohne dass Musicnet selbst
irgendetwas parsen oder filtern müsste.

### 1.3 Depends On Intents

* Keine direkten Abhängigkeiten - betrifft `DownloadPlaylistCommandBuilder`
  (`app/services/download_playlist_command_builder.rb`)

## 2.0 What

```gherkin
Feature: Downloads - Kompakte spotdl-Ausgabe

  Als DJ
  möchte ich beim Herunterladen von Tracks eine lesbare Ausgabe sehen
  statt einer riesigen Wand aus wiederholten Fortschrittsbalken-Textblöcken

  Scenario: Download einzelner fehlender Tracks nutzt die einfache Textausgabe
    When ein Download-Befehl für 1-10 fehlende Tracks gebaut wird
    Then enthält der Befehl das Flag --simple-tui

  Scenario: Playlist-Sync nutzt die einfache Textausgabe
    When ein Download-Befehl für einen vollen Playlist-Sync gebaut wird
    Then enthält der Befehl das Flag --simple-tui
```

## 3.0 How

### 3.1 Implementation Context

`DownloadPlaylistCommandBuilder#build_track_urls_command`/`#build_sync_command` bekommen beide das
zusätzliche Flag `--simple-tui`. Bestehende Tests, die auf den exakten Kommando-String prüfen,
werden entsprechend angepasst. Keine Änderung an `DownloadResultParser` nötig — der liest ohnehin
nicht die Konsolen-Ausgabe, sondern die `--save-file`/`--save-errors`-Dateien und
`Track#track_path` nach dem Lauf (siehe CLAUDE.md "Download flow").
