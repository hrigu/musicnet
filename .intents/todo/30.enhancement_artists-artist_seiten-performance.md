# 30 - Enhancement (Artists-Artist): Index und Detailseite ohne Query-Lawine rendern

## 1. Why

### 1.1 Objective

Die Künstler-Übersicht (`artists#index`) soll ihre Playlist-Badges gebündelt laden statt mit
einer Query pro Künstler, und die Künstler-Detailseite (`artists#show`) soll ihre Track-Tabelle
mit denselben Bündel-Ladungen rendern wie die Track-Übersicht.

### 1.2 Context

Log: `/artists` braucht 2.2–4.5s pro Aufruf mit **1802 Queries** (1797 Künstler + Overhead).
Ursachen:

1. `_artist.erb:7` ruft `artist.playlists_of_the_tracks` auf — eine eigene JOIN-Query pro
   Zeile (Badge-Spalte). Das sind die 1797 Playlist-Loads, die bei der Intent-27-Planung
   fälschlich der Track-Seite zugeordnet wurden.
2. `ArtistsController#index` lädt `includes(:albums, :tracks)` — `:albums` verwendet das
   Partial gar nicht (nur `tracks.length`), das Preload ist toter Ballast.
3. `artists#show` rendert `tracks/tracks` mit `@artist.tracks` **ohne** `includes` und ohne
   `Track.preload_track_paths` — pro Track eigene Queries für Artists/Album/Badges und seit
   Intent 29 ein Verzeichnis-Scan pro Zeile (Soundfile-Badge).

## 2.0 What

```gherkin
Feature: Artists Artist - Seiten gebündelt laden

  Als DJ
  möchte ich dass Künstler-Übersicht und -Detailseite ihre Daten gebündelt laden
  damit die Seiten schnell rendern.

  Scenario: Playlist-Badges im Index ohne Query pro Künstler
    Given mehrere Künstler, deren Tracks in Playlists liegen
    When ich die Künstler-Übersicht aufrufe
    Then sehe ich pro Künstler die Badges der Playlists seiner Tracks
    And die Playlists werden mit einer einzigen Query geladen

  Scenario: Detailseite wie die Track-Übersicht
    Given ein Künstler mit mehreren Tracks
    When ich seine Detailseite aufrufe
    Then wird jede beteiligte Tabelle mit genau einer Query geladen
    And die Track-Pfade werden mit einem einzigen Verzeichnis-Scan aufgelöst
```

## 3.0 How

### 3.1 Implementation Context

Muster von Intent 24/27, TDD über Request-Specs mit Query-Zählung
(`ActiveSupport::Notifications`) bzw. `Dir.children`-Zählung (Muster Intent 29):

1. **Index-Badges bündeln**: Klassenmethode `Artist.playlists_by_artist_id` — eine Query
   (`Playlist` join `playlist_tracks` → `track` → `artists`, distinct auf Playlist+Artist,
   `SELECT playlists.*, artists_tracks.artist_id`), im Controller zu einem Hash
   `artist_id => [Playlists]` gruppiert und dem Partial als Local übergeben.
   `artist.playlists_of_the_tracks` bleibt für Einzel-Kontexte bestehen.
2. **Ungenutztes Preload weg**: `includes(:albums)` im Index entfernen (`:tracks` bleibt
   für `tracks.length`).
3. **Detailseite**: Tracks in `artists#show` mit `includes(:artists, :album,
   playlist_tracks: :playlist)` laden und `Track.preload_track_paths` aufrufen
   (analog `tracks#index`); die Alben-Tabelle nutzt das bestehende `artist.albums`.
